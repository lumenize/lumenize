---
title: Using Resend instead
description: Configure @lumenize/auth to send via Resend rather than Cloudflare Email Sending
---

The default email provider for `@lumenize/auth` is [Cloudflare Email Sending](/docs/auth/getting-started#email-provider) — a native Worker binding, no API key, fitting the Cloudflare-first posture of this monorepo. But it requires the Workers Paid plan, and you may already use Resend elsewhere or want its 100-emails-per-day free tier.

**The provider is chosen by the environment, not by your code.** Your `AuthEmailSender` always extends `AuthEmailSenderBase`; to send via Resend, set `EMAIL_PROVIDER=resend` (or simply omit the `EMAIL` binding) and provide `RESEND_API_KEY`. Templates, subjects, the `AUTH_EMAIL_SENDER` service binding, and the `from`/`replyTo`/`appName` settings are all identical.

## When to choose Resend

- You want to stay on the Workers Free plan.
- Your organization already uses Resend for transactional email.
- You want Resend's deliverability dashboards or per-message analytics.

Otherwise, the default [Cloudflare Email Sending path](/docs/auth/getting-started#email-provider) is simpler.

## Setup

**1. Create a Resend account and verify your domain**

Sign up at [resend.com](https://resend.com), then [verify your sending domain](https://resend.com/docs/dashboard/domains/introduction). You'll add DNS records (SPF, DKIM, DMARC) — Resend's dashboard walks you through it.

**2. Generate an API key**

Create an API key in the [Resend dashboard](https://resend.com/api-keys) and add it to your environment:

```bash
# Local development (.dev.vars)
RESEND_API_KEY=re_...

# Production
wrangler secret put RESEND_API_KEY
```

**3. Create your AuthEmailSender class**

The same class as the default path — it extends `AuthEmailSenderBase` regardless of provider:

```typescript @check-example('packages/auth/test/for-docs/email-sender.test.ts')
import { AuthEmailSenderBase } from '@lumenize/auth';

export class AuthEmailSender extends AuthEmailSenderBase {
  from = 'auth@myapp.com';  // must match your verified Resend domain
}
```

Export this class from your Worker entry point alongside your other classes. `AuthEmailSenderBase` provides the default HTML templates and subject lines; the Resend transport (selected by env, below) handles delivery.

**4. Select Resend in wrangler.jsonc**

There's no `send_email` binding for Resend — it's a plain HTTPS API. Wire the self-referencing service binding, and set `EMAIL_PROVIDER` to `resend` so the choice is explicit (omitting the `EMAIL` binding also selects Resend, but an explicit `EMAIL_PROVIDER` avoids a silent fallback if the binding is ever dropped):

```jsonc
{
  "services": [
    {
      "binding": "AUTH_EMAIL_SENDER",
      "service": "your-worker-name",       // must match your wrangler.jsonc "name"
      "entrypoint": "AuthEmailSender"
    }
  ],
  "vars": {
    "EMAIL_PROVIDER": "resend"
  }
}
```

## Customizing templates

Template and subject overrides work identically to the default path — see [Customizing Email](/docs/auth/customizing-email). You extend the same `AuthEmailSenderBase`; only the environment differs.

## Switching providers later

Switching providers is an **environment change, not a code change**: add or remove the `EMAIL` binding, set or unset `EMAIL_PROVIDER`, add or remove `RESEND_API_KEY`. Your `AuthEmailSender` class never changes. No schema migrations, no template rewrites.
