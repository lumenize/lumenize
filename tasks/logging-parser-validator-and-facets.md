# Logging in ts-runtime-parser-validator & Nebula Facet Validation

**Status**: Plan decided 2026-06-02 — parent-side logging in the Nebula DOs.
The parser-validator package itself stays logging-free.

> Findings captured 2026-06-01 on branch `claude/mesh-nebula-logging-audit-q4P5U`.
> Plan section added 2026-06-02 after deciding between parent-side and
> facet-side instrumentation (see § "Why parent-side, not facet-side").

## Scope of the audit

- `packages/ts-runtime-parser-validator/src/`
- The Nebula code that *uses* the parser/validator as a per-version facet:
  - `apps/nebula/src/star.ts` (`#ensureFacet`, facet lifecycle)
  - `apps/nebula/src/resources.ts` (`transaction` → `facet.parseBatch`)

## What the facet path is

A "facet" here is a per-ontology-version `ParserValidator` Durable Object that
validates/parses resource data against a dynamically compiled schema.

- `star.ts:#ensureFacet()` loads the validator facet via
  `getParserValidatorFacet()` with a `bundleId` keyed on `${galaxyId}/${version}`,
  caches it in `#facet`, and recreates it when new ontology state is installed.
- `resources.ts:transaction()` receives the facet and calls
  `facet.parseBatch(requests)`. Results are a `Map` of
  `{ valid: true, data } | { valid: false, errors }`. Failures are collected
  into a `validationErrors` map and returned as `{ ok: false, errors }`.

## Findings — current logging state

1. **`@lumenize/ts-runtime-parser-validator` has zero logging.**
   - No `@lumenize/debug` import; no `console.*` calls anywhere in `src/`.
   - By design it **returns** validation errors as data
     (`{ valid: false, errors: [...] }`) and never throws at runtime. Only
     compile-time helpers (`extractTypeMetadata`, `generateParseModule`) throw,
     and only for invalid `@default` values / compile errors.
   - Consequence: the *only* place a decision to log can be made is the caller.

2. **Nebula's facet integration logs nothing — on success OR failure.**
   - `resources.ts` (~lines 310–326): validation failures are collected into
     `validationErrors` and returned to the client with **no server-side log**.
   - No log on the success path (no record of what was validated / which
     defaults were filled).
   - `parseBatch()` is **not** wrapped in try/catch — a facet load/RPC failure
     surfaces as a bare transaction error with no context (bundleId, ontology
     version, which resources).

3. **No logging of facet load/init in `star.ts:#ensureFacet`.**
   - No visibility into facet cache hit/miss, Worker Loader bundle creation,
     load success/failure, or timing.

## Why this matters (observability gap)

An operator cannot answer "why did this resource write get rejected?" or "which
type keeps failing validation, and on which field?" from server-side logs alone
— the information only exists in the client's error response. Facets are
compiled per ontology version, so the load/compile path is exactly where a
failure log would be most valuable.

## Why parent-side, not facet-side

The facet is a Worker Loader–compiled DO whose `ParserValidator` class is
machine-generated by `generateParseModule()`. Two real options were considered:

1. **Capability passing** — `loader.get(bundleId, async () => ({ ..., env: { LOG } }))`
   and inside the facet `this.env.LOG.warn(...)`. This is **structurally
   possible** (Worker Loader's `env` and `WorkerStubEntrypointOptions.props`
   support arbitrary capabilities, including RPC stubs and `RpcTarget` instances;
   Workers RPC supports passing functions). It does NOT contradict the "closures
   don't serialize" intuition — that was wrong for Workers RPC.
2. **Parent-side** — keep the facet logging-free; wrap the RPC call in the
   parent, log from there.

Picked (2) because every event the audit identified is observable from the
parent:

| Event | Observable from |
|---|---|
| Facet load — bundleId, version, cache hit/miss | Parent (facet doesn't know it's being loaded) |
| Facet load failure (compile error, missing module) | Parent (thrown from `loader.get(...)` setup) |
| Validation failure counts + sample error paths | Parent (returned `Map<id, ParseResult>`) |
| RPC failure during `parseBatch` | Parent (try/catch at the call site) |

No event identified by the audit lives **during** facet execution. If a future
need arises (per-request parse-timing, default-value substitution traces, etc.),
capability-passing is the documented escape hatch — `env.LOG` injected at
`loader.get()`. Until then, modifying `generateParseModule()` to emit logging
calls couples the package's "errors as data, no logging" design to a
Lumenize-specific contract for zero observability value today.

## Plan

The parser-validator package stays as-is. All work is in `apps/nebula/src`,
using the `nebula.{Class}.{method}` namespace convention from the sibling
nebula-logging task.

### 1. `star.ts:#ensureFacet()` — facet-load events

Detect cache miss by emitting a log inside the `loadModuleSource` callback
passed to `getParserValidatorFacet`. The callback only fires on the cold
Worker-build path; the warm path is the same-isolate cache lookup the helper
already does, so there's nothing to instrument on the warm side.

```typescript
const log = debug('nebula.Star.ensureFacet');
this.#facet = getParserValidatorFacet(
  this.ctx,
  this.env.LOADER,
  `${this.#galaxyId}/${row.version}`,
  () => {
    // Cache miss — first reference to this bundleId on this Worker project.
    log.info('facet cold load', {
      bundleId: `${this.#galaxyId}/${row.version}`,
      galaxyId: this.#galaxyId,
      version: row.version,
    });
    return row.validatorBundle;
  },
);
```

Levels:
- `info` for cold load (rare, operational milestone).
- No warm-path log (would fire on every transaction; no value).
- `error` for load failure surfaces naturally when the failure escapes
  `parseBatch` — covered in step 2 (no separate try/catch needed in
  `#ensureFacet` itself).

Same edit applies to the second `getParserValidatorFacet` call in
`#installState` (~line 140). Identical block.

### 2. `resources.ts:transaction()` — RPC boundary + validation outcomes

Wrap the `await facet.parseBatch(requests)` call (~line 317) in try/catch. Log
RPC/load failures with full context, then re-throw. Inspect the returned `Map`
for validation failures and log a structured warn.

```typescript
const log = debug('nebula.Resources.transaction');
let results;
try {
  results = await facet.parseBatch(requests);
} catch (err) {
  log.error('facet.parseBatch threw', {
    galaxyId,                                      // already in scope
    version: row.version,                          // available from #ensureFacet
    bundleId: `${galaxyId}/${row.version}`,
    requestCount: requests.size,
    typeNames: [...new Set([...requests.values()].map(r => r.typeName))],
    error: err instanceof Error ? err.message : String(err),
  });
  throw err;
}

for (const [resourceId, result] of results) {
  if (result.valid) {
    const op = ops[resourceId];
    if (op.op === 'create' || op.op === 'put') op.value = result.data;
  } else {
    validationErrors[resourceId] = { type: 'validation', errors: result.errors };
  }
}

const failureCount = Object.keys(validationErrors).length;
if (failureCount > 0) {
  log.warn('validation failures', {
    count: failureCount,
    requestCount: requests.size,
    sampleErrors: Object.entries(validationErrors).slice(0, 3).map(
      ([id, e]) => ({ resourceId: id, errors: e.errors }),
    ),
  });
}
```

Levels:
- `error` for the thrown-RPC path — system failure, always output.
- `warn` for validation failures — operational/filterable; client error, not a
  bug. Don't `error` here: a high rate of malformed client input would flood
  always-output logs.
- No success-path log — high volume, low signal. The transaction already returns
  a structured result the caller sees.
- The `Resources` class will need access to `row.version`/`galaxyId`; if not
  already passed in, surface them via the existing constructor injection used
  by `Star`. (Implementation detail — check the actual signature when writing
  the change.)

### 3. What does NOT change

- `packages/ts-runtime-parser-validator/src/` — no logging added; no
  `@lumenize/debug` dependency; the package's "errors as data, never throws at
  runtime" design is preserved.
- Generated module from `generateParseModule()` — unchanged.

### 4. Documented escape hatch (for future)

If/when per-request timing or default-application tracing is wanted, the
mechanism is `env.LOG` injection at `loader.get(...)` setup, with `LOG` being
either an RPC stub to a logger DO or an `RpcTarget` instance. Capture this in
the package's docs only when there's a concrete use case — premature
documentation invites speculative coupling.
