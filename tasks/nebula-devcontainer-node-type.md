# LumenizeContainer / NebulaContainer — the 4th Lumenize node type

**Status**: Design complete (reviewed: framing + conformance 2026-06-17; re-reviewed post-spike 2026-06-17) — **gate CLEARED: the agent-channel spike RAN + GOed 2026-06-17** (`tasks/spike-container-agent-channel.md`; Q1 mechanism + Q2 deployed-latency both ✅, ~30 ms warm channel edge-local; memory [[agent-channel-container-exec]]). **Now build-ready** (next in build order). **Landing this task satisfies the remaining half of ADR-007's acceptance gate** (the Q5 spike was the other half) → flip `docs/adr/007-shared-node-security-core.md` Proposed→Accepted on landing.
**Phase**: Studio container-vite pivot — foundation (`tasks/nebula-studio.md` § *UI-build architecture*).
**App**: `packages/mesh` (`LumenizeContainer` — **`@cloudflare/containers` approved as a mesh dep 2026-06-17**; mesh is already CF-coupled, this sits beside `LumenizeDO`/`LumenizeWorker`) + `apps/nebula` (`NebulaContainer`). Realizes **ADR-007 (Proposed)**. **Export ONLY via a `@lumenize/mesh/container` subpath** (decided — see Decisions), keeping core `@lumenize/mesh` container-free.

**Why this exists:** the pivot's dev preview fronts a vite container; its **reason for existing is using containers**. But to talk to the rest of Nebula it must be a **mesh node** — provide the comms+guards core: `lmz.call()` (receive), `lmz.ctn()` (outgoing), `onBeforeCall` + `@mesh()`. It extends `@cloudflare/containers` `Container` (which extends `DurableObject`), so it **cannot** inherit `LumenizeDO` — and shouldn't: it's **its own node type**, not a `LumenizeDO`. The DO ancestry (via `Container`) is **incidental**. (The Studio-specific `DevContainer` — vite + the preview proxy — is a `NebulaContainer` consumer built later in the #1a dev-loop reshape; this file builds only the node type.)

## Scope of the node core (narrow — per ADR-007)
- **IN (the shared core, composed):** `lmz.call()` receive, `lmz.ctn()` outgoing continuations, `onBeforeCall`, `@mesh()`. This is exactly what **`LumenizeWorker`** composes — verified: `LumenizeWorker` has `onBeforeCall` + `executeEnvelope` and **no** `onStart`/`onRequest`/`alarm` (`lumenize-worker.ts`).
- **AVAILABLE, not part of the invariant:** `lmz.sql` (the node is DO-based via `Container`'s `ctx.storage`) for the node's own storage needs — but `ctx.storage` is **shared with `Container`'s own lifecycle state** (see *Storage namespace* below).
- **OUT — `Container` exclusively owns these; the mesh core takes none:** `Container`'s constructor `blockConcurrencyWhile`s `scheduleNextAlarm()` + `CREATE TABLE container_schedules` and its `alarm()`/`onStart()` are load-bearing for container lifecycle. So **`alarm`/`onStart` are not "available later" — they are owned.** A future Lumenize alarm need on this node type is **blocked**: route it through `Container.schedule()` or a sibling Worker, **never** `svc.alarms`. (Note the node thus **inherits Container's live `alarm`/`onStart`** — they are present and load-bearing, not absent; the mesh core just doesn't touch them — see m7.) The node overrides `Container`'s `fetch()` to pin the public port (M1), not a DO `onRequest()`.
- **EGRESS posture (name it; don't inherit the dangerous default).** `Container`'s `enableInternet` defaults to **`true`** (`container.js:325`) = open outbound. This node fronts a vite container (npm/registry traffic) and ultimately runs **agent-authored code** (#1a) — an SSRF/exfil surface if left open. **Safe default to pin: `enableInternet = false`,** opened only via an explicit `allowedHosts` allow-list or routed through the established `EgressBroker`/`globalOutbound` choke point ([[project_nebula_outside_world]]'s D2). Full egress hardening for the agent-code path is **#1a's** (named here so it isn't orphaned — like the exec-channel hardening, m4), but the *default-off* posture is a node-core decision and belongs here.

## Scope-isolation boundary (M1 — state it explicitly, don't let #1a inherit an unstated assumption)
`onBeforeCall` (the tenant scope guard) runs **only on the mesh path** (inside `executeEnvelope`). It does **not** cover `Container.fetch()`/`containerFetch`. That is **by design, not a hole** — but the public surface must be **pinned, not inherited:**
- **`fetch()` is the public vite-proxy surface** — it serves only the **public preview shell** (frontend code + ontology types every client gets anyway), exactly like DevStar's intentionally-open `onRequest` (#1a). No tenant data flows through it.
- **⚠️ The base `Container.fetch()` is NOT a safe public surface as-is (verified, M1-conformance).** `Container.fetch()` (`container.js:990`) honors an inbound **`cf-container-target-port`** header and forwards to *any* named port via `containerFetch(req, port)`; absent the header it uses `defaultPort`. So inheriting it unchanged means a public request carrying `cf-container-target-port: <command-port>` **reaches the command server** — the agent-channel spike's "no browser route to 9000" held only because its client never sent that header. **Invariant (pin in the node core):** `NebulaContainer`/`LumenizeContainer` MUST **override `fetch()`** to forward only to the preview port and to **strip/reject inbound `cf-container-target-port`**; the command-server port must **never** be in a publicly-reachable `defaultPort`/`requiredPorts`. Reachable command access stays exclusively on DevContainer's internal `containerFetch(req, <cmd-port>)`.
- **Everything sensitive — data, and the agent command channel — goes over the mesh** (`lmz.call`, `onBeforeCall`-gated). The agent talks to DevContainer over the mesh; the command-server port is reachable **only** by DevContainer's internal `containerFetch`, never via the public `fetch()` (the Q5 trust boundary — now *enforced* by the override above, not assumed).
- **Tests — split by runnability:** (i) *in-process, capable-of-failing:* with `fetch()` open, a cross-scope **mesh** data call is still rejected by `onBeforeCall`. (ii) *the port-not-reachable half:* a public `fetch()` carrying `cf-container-target-port: <command-port>` must **not** reach the command server — capable-of-failing by reverting the strip/pin. Real `containerFetch`/port behavior isn't exercisable in vitest-pool-workers (see Phase 2), so the *live-container* form of (ii) is an `it.skip` + named #1a/e2e blocker; do **not** assert it against a stub (the override-strips-the-header logic itself *is* unit-testable on the request object without a live container).

## The composable seam (corrected — M1 framing-stage finding)
Composed, not inherited: `LumenizeDO` + `LumenizeWorker` realize the receive path via `executeEnvelope` behind the `EnvelopeExecutorNode` seam. **`LumenizeClient` does NOT use `executeEnvelope`** — it hand-rolls the receive path (`#handleIncomingCall`). So the seam is shared by **2 of 3** today; the Client is a documented parallel path. `LumenizeContainer` composes the **DO-flavored** seam.

**The full receive contract a non-`LumenizeDO` base must implement** (m1 — not just "lmz/onBeforeCall/__executeChain"): public **`__executeOperation(envelope)`** (calls `executeEnvelope(…, { includeInstanceName: true })` — the surface `lmz.call` dispatches to), `lmz` with a working **`__init`**, **`callContext`** (the ALS-bound `this.lmz.callContext` — see below), `onBeforeCall`, `__executeChain` (**the secure path — `requireMeshDecorator` defaults true**, lmz-api.ts:905; the bypass `__localChainExecutor` is for the result-handler path only and must NOT be wired onto the inbound dispatch), and `__localChainExecutor`. Composition point: **`createLmzApiForDO(this.ctx, this.env, this)`** (n2); identity persistence relies on `Container`'s `ctx.storage` being SQLite-backed → register the class with **`new_sqlite_classes`** (durable-objects.md).

**`callContext` is REUSED, not reimplemented (confirmed 2026-06-17).** The ALS plumbing is shared: `executeEnvelope` runs the chain inside `runWithCallContext(envelope.callContext, …)` (lmz-api.ts:900), and the `this.lmz.callContext` getter resolves it via the shared `requireCurrentCallContext()` helper (lmz-api.ts) — used by **both** the DO and Worker factories (a previously-duplicated getter was extracted into that helper during this review). So `LumenizeContainer`, composing `createLmzApiForDO`, gets `callContext` **for free**; do not hand-roll it. The browser `LumenizeClient` is the documented divergence (no AsyncLocalStorage → reads a synchronously-captured `#currentCallContext` field) — already a required Phase-1 divergence row ("no `runWithCallContext`/ALS").

**Coexistence constraint (the load-bearing rule behind m2, m7 — pin it):** `Container`'s constructor runs its own lifecycle setup inside `blockConcurrencyWhile` and owns `onStart`, so the usual "set up in `onStart`" DO escape hatch is **gone**. `LumenizeContainer` therefore contributes **zero constructor body and no `onStart` override** — identity composes purely through the **lazy `lmz` getter** (`createLmzApiForDO`, like `LumenizeWorker`/`LumenizeDO`'s getter), so `__lmz_do_*` writes land on the **first inbound mesh call**, after `Container`'s lifecycle is already up. The mesh core also **does not redefine `alarm`/`onStart`/`fetch`-lifecycle** — it inherits `Container`'s (alarm/onStart) and overrides only `fetch()` (M1).

## Decisions pinned
- **`class LumenizeContainer extends Container`** in `packages/mesh` — a **concrete base** composing the comms+guards core (default; one consumer, YAGNI). A generic `withLumenizeMesh(Base)` mixin is **deferred** unless a 2nd third-party base appears.
- **Subpath export (footprint hygiene, decided — not "decide in build"):** `LumenizeContainer` is exported **only** via `@lumenize/mesh/container` (separate entry; `@cloudflare/containers` as that subpath's dep), so core `@lumenize/mesh` stays container-free — it's MIT and imported by `auth`/`fetch`/`testing`/`ts-runtime-parser-validator`, and a top-level export would pull `@cloudflare/containers` into all of them (workflow.md "smallest built footprint"). Precedent: the `./client` subpath.
- **`requireAdmin` param (nit):** `requireAdmin` types its param `NebulaDO` but only reads `instance.lmz.callContext`; `NebulaContainer` is a *sibling*, not a `NebulaDO`. Widen the param to a shared structural shape (`{ lmz: { callContext: … } }`) both node types satisfy rather than casting `NebulaContainer` to `NebulaDO`.
- **`NebulaContainer extends LumenizeContainer`** — `onBeforeCall` reuses `buildAuthScopePattern` + `matchAccess` (mirror `NebulaDO`'s guard; a **sibling**, not a subclass, of `NebulaDO`).
- **Addressing contract (M3):** a DevContainer is **always** addressed by its `parseId`-valid tenant-scoped name **`{u}.{g}.dev`**, never a 64-hex `idFromString` — so `buildAuthScopePattern(name)` yields exactly the scope the caller must already hold (the name==routing-key soundness that makes the tier-DO guard work; `nebula-do-scope-isolation.md` M4). Enforced by a fail-closed test, not assumed.
- **Storage namespace (m2):** `createLmzApiForDO` persists identity (`__lmz_do_*`) into the same `ctx.storage` `Container` uses. Two reserved names in two namespaces: the **kv key** `OUTBOUND_CONFIGURATION` (`ctx.storage.kv`, `container.js:1066`) — the real collision surface, since `__lmz_do_*` is kv-only — must not be reused for identity; and the **SQL table** `container_schedules` (`CREATE TABLE`, `container.js:390`) — don't create a Lumenize table by that name.
- JWT stays at the Gateway; DO-side ALS `callContext` path. No raw RPC for the node's own mesh calls (mesh.md, ADR-003).

### Phase 1 — Divergence inventory + the composable seam (no consolidation)
- Produce the **inventory** (build-now): `grep` every type calling `executeEnvelope` (`packages/mesh/src`) **plus** the Client's `#handleIncomingCall`. **Artifact-shaped criterion (m6):** for each `EnvelopeExecutorNode`/receive requirement, state per existing type whether it's satisfied via `executeEnvelope` or hand-rolled, and which `LumenizeContainer` reuses verbatim vs. supplies. Pin the **four Client divergences** as required rows: no `__init`, no `runWithCallContext`/ALS, no `{$error}` wrap, response-via-WS-message.
- Enumerate `Container`-owned keys/tables as **reserved** — at minimum the SQL table `container_schedules`, the kv keys `OUTBOUND_CONFIGURATION` and **`__CF_CONTAINER_STATE`** (`ContainerState`, `container.js:193`), and the alarm slot; **audit `ContainerState` + the `Container` source so the list is provably exhaustive**, not a hand-pick. Document "how to compose the comms+guards core onto a base" + the no-collide namespace rule.
- **Do NOT consolidate the Client's path** — deferred unless it blocks composition.
- **Success:** the inventory artifact above exists + names what `LumenizeContainer` supplies; existing DO/Worker/Client tests stay green.

### Phase 2 — `LumenizeContainer` (mesh layer)
- `extends Container` + composes the receive contract (above); `lmz.sql` available; **`fetch()` overridden to pin the public port** (M1); no `onRequest`.
- **Deployability:** registration needs **both** `new_sqlite_classes` (the DO-class registry) **and** a `containers` block (`image`/`class_name`/`instance_type`) in `wrangler.jsonc`. Prod `apps/nebula/wrangler.jsonc` currently has **neither** — note this so a deploy/build doesn't discover it late.
- **⚠️ Test-feasibility precheck (do FIRST):** `Container`'s constructor **throws if `ctx.container === undefined`** (`container.js:350`), which vitest-pool-workers populates only when a `containers` engine config is wired. So **start Phase 2 with a one-line spike:** does `class X extends Container {}` construct under pool-workers (test wrangler supplying a `containers` block)? **Pin the chosen path before writing the suite:**
  - **If it constructs** → run the in-process seam/guard tests below against the assembled `LumenizeContainer`/`NebulaContainer` (test wrangler carries `containers`).
  - **If it throws** → test the **composed seam directly** (`createLmzApiForDO(ctx, env, node)` + `executeEnvelope` on a minimal non-`Container` harness node that mirrors the receive contract), and defer the *assembled* `extends Container` construction + lifecycle coexistence to a **deployed e2e** (`it.skip` + named blocker), like the `containerFetch` carve-out. Either way the seam logic stays covered.
- **Success — mesh seam (M4):** an inbound `lmz.call` **lands via `__executeOperation`** and returns; `onBeforeCall` fires — **mutation-check by commenting out `node.onBeforeCall()` in `executeEnvelope` (lmz-api.ts:902), confirm the marker-count test goes RED, restore** (m8; drop the `T-local-skip` mirror — its negative half needs `svc.alarms`, which this node lacks). The **outgoing `lmz.ctn`** sub-test runs **after** an inbound call has stamped identity (outgoing `call`/`ctn` needs `this.lmz.bindingName`, set only by `executeEnvelope`'s `__init` on first inbound — lmz-api.ts; the node has no `__initFromHeaders` path); add the **negative micro-check** that an outgoing `ctn` on a never-called instance throws the expected "doesn't know its binding name" error (makes the ordering explicit + capable-of-failing).
- **Success — `@mesh` enforced (security-relevant):** a mesh `lmz.call` to a **plain (non-`@mesh`) method is rejected** ("@mesh decorator required"). Mutation-check: flip the composed `__executeChain` to `requireMeshDecorator: false` (the `__localChainExecutor` bypass setting) → the call lands → RED. Guards against a miscomposition that wires the bypass executor onto the inbound dispatch and silently exposes every method.
- **Success — error round-trip (m9, ADR-002):** a `@mesh` method that throws a custom `Error` surfaces to the `lmz.call` caller with `name` + custom own-properties intact (assert by `err.name` + property presence, **not** `instanceof` — mesh.md) — exercises the `{$error}` wrap unique to `executeEnvelope`.
- **Success — narrow-core / no-collision (m7, reframed — the old `!Object.hasOwn` was vacuous):** `alarm`/`onStart` live on `Container.prototype` (one level up), so `!Object.hasOwn(LumenizeContainer.prototype, 'alarm')` is true regardless of what the core does — it proves nothing and *misstates* reality (the node **inherits a live** `alarm`/`onStart`). Assert instead that the core left them **untouched**: `LumenizeContainer.prototype.alarm === Container.prototype.alarm` **and** `…onStart === Container.prototype.onStart` (capable-of-failing: a careless `prototype.alarm = …` / `svc.alarms` wiring breaks the identity); plus the **positive own-prop control** that the core DID add `onBeforeCall`/`__executeOperation` (`Object.hasOwn` true). Pin `@cloudflare/containers` **v0.3.7** as the baseline so upstream drift breaks it.
- **Success — lifecycle coexistence concrete (m2):** after constructing the node and firing **one inbound mesh call**, both the Lumenize identity kv key (`__lmz_do_instance_name`) **and** `Container`'s `container_schedules` table exist and the alarm slot is intact — proves identity (composed via the lazy `lmz` getter, after the base lifecycle) doesn't clobber `Container`'s startup. (Runs in whichever harness the precheck selected.)
- Real `containerFetch`/port behavior is **not** exercisable in vitest-pool-workers (no OS container) — defer to a deployed e2e or the agent-channel spike via `it.skip` + named blocker (M4); do **not** assert it against a stub.

### Phase 3 — `NebulaContainer` (nebula layer)
- `extends LumenizeContainer`; `onBeforeCall` scope-isolation via `buildAuthScopePattern`/`matchAccess` (mirror `NebulaDO`).
- **Success (mirroring `scope-isolation.test.ts`):**
  - A **genuinely-minted** (not forged) cross-scope / non-admin caller is **rejected** (`Error`, B1) — give `NebulaContainer` **one trivial guarded `@mesh` write** (e.g. an `lmz.sql` INSERT) so "writes nothing" is capable-of-failing (mutation-check: comment out the `matchAccess` reject → the write lands) (m5). An in-scope caller passes.
  - **Fail-closed addressing (M3):** a `NebulaContainer` addressed with a foreign-scope or unparseable/hex name is rejected (mirror `T-malformed`).
  - **B5 (m5):** extend the frozen non-admin-`@mesh` allow-list with a walk/`it` over `NebulaContainer`'s prototype (the dynamic partition by `getMeshGuard(m) !== requireAdmin`), so a new non-admin method fails the test.
