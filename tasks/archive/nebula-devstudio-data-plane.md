# Nebula — DevStudio data-plane (Child 1 of the multi-user chat thread)

**Status**: **DONE + ARCHIVED 2026-06-29.** All 5 phases built + green (pool-workers 537/537, browser 4/4, ui-smoke 4/4; tsc clean); verifier panel PASS (0 blockers/majors); nuggets extracted into `nebula-pre-alpha.md`. Frozen on archive.
**Parent**: [`nebula-pre-alpha.md`](nebula-pre-alpha.md) § Current focus (the 3-child chat thread).
**Siblings (do NOT start until this lands + archives)**: Child 2 = [`nebula-query-subscriptions.md`](nebula-query-subscriptions.md) (parent:child query subs, built againts the capability this child extracts); Child 3 = `reactive-ai-chat` (write after Child 2).

## Objective

- **What:** make **DevStudio** able to host **Resources** (with DAG permissions + single-resource subscribe) the same way **Star** does today.
- **Why:** so the Studio chat can later model each turn as a `Turn` Resource FK'd to a `Session` parent (Child 3).
- **Mechanism — extraction, not duplication:** lift Star's resource data-plane into **one composable capability** (ADR-007 — composition, never reimplemented) that both `Star` and `DevStudio` compose.
- **Proof:** end-to-end with a **real `NebulaClient`** doing CRUD + single-resource subscribe against DevStudio, with **Star's existing suite still green** as the faithfulness net.

**Out of scope (explicit):**
- **Parent:child query subscriptions** → Child 2.
- **The chat-loop rewrite / turn relocation / chat-hang fix** → Child 3.
- **`subscribeReload` / `ReloadSubscriptions` on DevStudio** — this is the **dev-preview bundle-refetch channel** (the publish-refresh signal telling a live app preview to re-fetch its bundle; star.ts:567-624, reload-subscriptions.ts), a `.dev` Star / preview concern — **NOT** an ontology mechanism. DevStudio's chat clients get `Session`/`Turn` updates through the ordinary resource-subscribe channel, so there is no reload channel to host here. (Orthogonal to the fixed-ontology point.)
- **`TreeSubscriptions` (org-tree subscribe) on DevStudio** — not needed for Child 1 (single-resource subscribe suffices for CRUD). Child 3 *may* revisit it for a live participant/grant view, but not this child.

## What exists today (grounding — verified 2026-06-29)

- `Star extends NebulaDO` wires the whole data-plane in `onStart` by **constructor injection** ([star.ts:44-62](../apps/nebula/src/star.ts)): `DagTree(ctx, ()=>callContext, onChanged)` → `Resources(ctx, ()=>callContext, dagTree)` → `Subscriptions(ctx, ()=>callContext, dagTree, resources)`.
- Star's **resource mesh surface** splits into **Handler 1 / Handler 2** (the code's own naming, `star.ts:358`): the `@mesh()` wrappers `transaction` / `read` / `subscribe` (**Handler 1** — client-origin plumbing + the `#isCachedVersion(appVersion)` → `OntologyStaleError` version-gate) dispatch to the local handlers `doTransaction` / `doRead` / `doSubscribe` (**Handler 2** — the actual op + `#broadcast`/`onMutations` fanout via `svc.broadcast` + result-delivery to `NEBULA_CLIENT_GATEWAY`). Plus `@mesh()` `unsubscribe` / `dagTree` (single `@mesh` entry for the DagTree API) and `onBeforeCall` (aud-lock + first-admin seed) ([star.ts:260-690](../apps/nebula/src/star.ts)). **The version-gate is Galaxy-multi-version-specific — DevStudio's fixed ontology never triggers it (D8).**
- **Ontology source is the only Galaxy coupling.** `Resources.transaction(ops, version, newETag, facet, …)` takes the compiled `facet` **per call** ([star.ts:389](../apps/nebula/src/star.ts)) — so `Resources`/`Subscriptions` are already **provenance-agnostic**. The Galaxy-fetch/cache lives entirely in Star's `#ensureFacet` / `#row` ([star.ts:147](../apps/nebula/src/star.ts)). This is the seam.
- `DagTree` is also **Star-only** (`new DagTree` appears once, star.ts:45). It extracts with the rest.
- `DevStudio extends NebulaDO` ([dev-studio.ts:115](../apps/nebula/src/dev-studio.ts)), addressed at `{u}.{g}.dev` (co-named with the `.dev` Star on the `STAR` binding). It already mounts a **platform-fixed typia facet from a constant** via `#ensureToolArgsFacet()` → `generateParseModule(TOOL_ARGS_TYPES)` ([dev-studio.ts:447](../apps/nebula/src/dev-studio.ts)) — the exact pattern DevStudio's `Session`/`Turn` ontology provider uses.
- **Compile vs mount (the ts-runtime-parser-validator split — answers "where does the compile go").** The validator is produced by the **pure exported helper** `compileOntologyVersion` → `extractTypeMetadata` + `generateParseModule(writeShape, relationships)` ([galaxy.ts:109-125](../apps/nebula/src/galaxy.ts)), which **Galaxy** runs **eagerly** at `appendOntologyVersion`, storing the `validatorBundle` *source* in the version row; **Star only mounts** that precompiled bundle via the Worker Loader (`#ensureFacet` → `getParserValidatorFacet`). The helper is **host-agnostic** — DevStudio calls it directly on its in-source `Session`/`Turn` types (it has no per-galaxy registry to route through), lazily mounted via the Worker Loader exactly like `#ensureToolArgsFacet`. **The compile runs on DevStudio, NOT Galaxy, NOT at build time** (see D2).

## Phases

### Phase 1 — Extract the data-plane into a composable capability; Star delegates
**Goal**: The capability is the single home for `DagTree`+`Resources`+`Subscriptions` and **Handler 2** (`doTransaction`/`doRead`/`doSubscribe` + `#broadcast`/`onMutations` + result-delivery), parameterized by the D2 `getOntology()` provider. Star keeps its **Handler-1** `@mesh` wrappers (incl. the version-gate, D8) but they now dispatch into the capability; Star's `#ensureFacet` becomes Star's `getOntology()` provider impl (D6 lists what stays on Star).

**Success Criteria**:
- [x] Star's **existing** test suite green, unchanged in behavior (regression net) — `npx vitest run` in `apps/nebula`.
- [x] Capable-of-failing: permission-denied write and ontology-validation-failure still reject through the capability (a test that reddens if either guard is dropped in the move).
- [x] Capable-of-failing: a **not-found node** and a **permission-denied** deliver DISTINCT typed errors through the capability's result-delivery — assert `err.name` for each (`NodeNotFoundError` vs `PermissionDeniedError`; detect by name+property, not `instanceof`, per [mesh.md]) — so Handler 2's broadened catch doesn't conflate a malformed request with a permission failure (review m6, [security.md]). Mutation-verified: gutting each operand's catch (flattening the delivered Error) reddens that operand's assertion.
- [x] No code in the capability / `Resources` / `Subscriptions` reaches into Galaxy directly — the injected `getOntology()` is the **only** ontology entry. Verify: `grep -rnE 'GALAXY|getLatestOntologyVersion|getOntologyVersion|OntologyState' <capability-file> resources.ts subscriptions.ts` → **no hits** (D6's `#ensureFacet`/`#installState`/`setOntology` stay on Star).
- [x] `Star.prototype` `@mesh`-surface-freeze test (`test/test-apps/baseline/dev-star-data-lifecycle.test.ts` — "the admin-gated set equals the frozen allow-list") still passes **unchanged** (Handler 1 stays on Star, so Star's surface is preserved).

## Notes / risks
- **The whole point for Child 2**: build the capability so QuerySubs slots in as one more constructor-injected unit (like `Subscriptions`), and the per-push permission recheck / DagTree batch-eval live in the capability, **never** in a `Star.#broadcast` private — so Child 2 lands on Star + DevStudio at once.
- **`svc.broadcast`** is a `LumenizeDO` framework primitive, so DevStudio (also a `LumenizeDO`) can fan out the same way Star does — but per D1/m1 the continuation construction stays host-side (the capability calls host-bound thunks), so the `onMutations → svc.broadcast` wiring does NOT "move cleanly" verbatim.
- **Stale-read gating is subscribe-time-only today** (`Star.#broadcast` does NOT re-check DAG read permission per push — "accepted for demo", star.ts:660-663). Lifting `#broadcast` into the shared capability carries that behavior to the chat data plane. **Accepted as-is for Child 1** (it ships no user-visible answer surface) — but a fenced handoff: **Child 2 closes it** (its Open Q2 — per-push permission recheck on the single-resource fanout) and **Child 3's governed AI-chat surface must not carry user-visible answers until that lands** ([security.md]/strategy: the chat answers only from data the user may see; a revoked participant must stop receiving pushed `Turn`s). Mutations stay correctly gated (resources.ts) — the gap is reads/pushes only. (Review m3.)
- **Lifecycle reminder** (why DevStudio, not the `.dev` Star): `Session`/`Turn` use a platform-fixed ontology that must survive app-data wipes; the `.dev` Star's Resources carry the user-developer's wiped/regenerated ontology. Do not be tempted to reuse the `.dev` Star. **Sentinel-version contract (review n1):** the fixed version is stamped into every snapshot's `meta.ontologyVersion` (resources.ts), so editing the `Session`/`Turn` types is a **breaking change** — it requires a sentinel-version bump **and** a wipe of DevStudio's resource snapshots (the same bargain as the `.dev` Star's `resetDevData`); there is no Galaxy version-registry / migration chain for DevStudio's fixed ontology.
- **Considered & declined for Child 1 — moving the *user-app* ontology compile off Galaxy into Star** (Star compiles `ontology.d.ts` on first init, then stores/mounts; Galaxy ships just the app file tree; the capability becomes fully self-contained). **Declined only because the current Galaxy-compiles/Star-mounts path works and is out of Child 1's scope — "if it's not broke, don't fix it"**, NOT on tsc-cost grounds (the compile is a one-time, post-ontology-change startup cost that's GC'd after — the tsc bundle doesn't really factor; Galaxy-compile-once is a minor optimization, not load-bearing). **Star-compile is arguably the *cleaner* end-state and stays a viable later refactor.** Orthogonal to Child 1's actual decision (Session/Turn compiled on DevStudio, which already loads tsc).
