## Goal

The conversational experience where a user-developer describes what they want, the agentic development engine generates their product (ontology + UI), and it runs live with real access control. Studio is the demo's wow moment вЂ” "I want to build X" в†’ working app on screen, then iterate. The user-developer never opens a code editor: **describe в†’ generate в†’ preview (repeat)**.

> Demo-narrative storyboard is deferred until the generation loop is validated end-to-end вЂ” but it's a **cold start** (we may edit the video to trim any long waits).

## The participant model (build target)

Per `nebula-dev-flows.md`, the cast is **DevStudio** (server DO вЂ” orchestrates the agentic loop, sole writer + source-of-truth via a shell `Workspace` + local git), **DevContainer** (a disposable Cloudflare Container running real `vite`/HMR вЂ” DevStudio pushes source to it via `applyChanges`), **Studio UI** (the chat SPA, served from **Workers Assets**), and the **Preview app** (the running generated app, embedded in an iframe in Studio UI). Dev data is a plain **`Star` at the `{u}.{g}.dev` instance** (no `DevStar` class вЂ” *the target state, not yet built; see the вљ пёЏ below*). All nodeв†”node edges are mesh.

## DevContainer dev loop (build)

The dev-loop *flow* is Flow 1 / 1c; the build specifics:

- **`DevContainer extends NebulaContainer`**, bound `DEV_CONTAINER`, addressed at the `{u}.{g}.dev` instance (`generateBindingVariations` keeps it disjoint from `STAR`); a smart-match guard routes `*.*.dev` shell requests to it. Two ports: **`:5173` vite** (public, ungated вЂ” shell + HMR) and **`:9000` command-server** (**host-DO-only** вЂ” `exec`/`writeFile`/`viteControl`/git). `fetch()` is a three-way branch (WS verbatim / shell-buffer+inject / asset stream) and **strips `cf-container-target-port`** so the public path can never reach `:9000`.
- **Scope-injection choke point:** DevContainer injects the **server-derived** `activeScope`/`authScope` (from `lmz.instanceName`, never the request) into the shell HTML as `<meta name=nebula-scope>` вЂ” the wrong-Star footgun guard.
- **Command methods carry `@mesh(requireAdmin)`** (not bare `@mesh` вЂ” `NebulaContainer.onBeforeCall` proves tenant scope but never `access.admin`, and `<id>.*` widening admits descendant non-admins). DevStudio invokes them as **awaited `lmz.callRaw`** (single-hop result-bearing transport, ADR-003 вЂ” never raw Workers RPC); `containerFetch` is used only *inside* DevContainer to reach its own `:9000`. **The path-traversal guard lives on the receiving in-container command-server** (where the bytes land вЂ” reject `..`/absolute, write nothing), with the DO `@mesh` method as defense-in-depth (`security.md` receiver-re-validates). `exec` is host-DO-only by construction (port stripped), so no further arg-sanitization for the demo.
- **Two entrypoint edits:** (M3) add `DEV_CONTAINER` to the serving-target switch (`entrypoint.ts` `isServingTarget`); (M2) allow the HMR WebSocket via a distinct `onBeforeConnect` touch-point (ungated, like the shell). *(The gate's `|| env.DEV_STAR` disjunct is dropped in **Phase 4**, when the in-DO serve it gated is deleted вЂ” don't half-collapse it here.)*
- **Deps are baked** into the container image (curated UI-lib set вЂ” DaisyUI, Lucide [ISC], etc.) в†’ **zero `npm install` on cold boot** (CF containers have no persistent volume, so baking is the only durable store; `standard-1`/ВЅ-vCPU clears the `vite build` starvation bar). Cold boot = boot + DevStudio re-pushing the full source tree (Flow 1c).
- **vite fully owns SFC compile** вЂ” the Star never compiles, there is no in-DO compile machinery. vite diagnostics come back as method-return data over the command channel for the agentic loop to read.
- **Deploy:** local container-image push works via **Docker Desktop + WARP** (`[[cf-container-deploy-proxy]]`); CI workflow is the headless fallback.

**Test strategy (vitest-pool-workers reality, `[[container-no-construct-pool-workers]]`):** `extends Container` can't construct under vitest-pool-workers, so test the composed seam via non-Container harnesses + pure prototype tests вЂ” the entrypoint serving-target gate (inert stub-DO bindings), the HMR-WS allow gate, the scope-injection as a pure derivation, and the `writeFile` path guard (**enumerate negatives**: a `../`-traversal path **and** an absolute path each *rejected + writes nothing*, plus an in-tree positive control вЂ” each capable-of-failing by neutralizing the guard). The assembled-construction e2e is an `it.skip` run with `wrangler dev` on **Docker Desktop** (the Container runs locally there via WARP; it just can't construct under pool-workers). A fully *deployed* e2e (for alpha testers) gates on the first full `apps/nebula` Worker deploy.
- **Deployed-e2e admin setup (no prod backdoor).** The deployed e2e must provision admin to exercise `@mesh(requireAdmin)`, but a deployed-Worker harness has no `miniflare.bindings` channel вЂ” so it uses a **separate deployed test-harness wrangler** carrying any bootstrap/test-mode flag with the documented-exception comment. **Production `apps/nebula/wrangler.jsonc` must never gain an admin/bootstrap/test-mode `var`** (it currently holds only `PRIMARY_JWT_KEY`/`NEBULA_AUTH_REDIRECT`/`LUMENIZE_APPROVED_ORIGINS` вЂ” keep it that way).

### Dev-data reset (`resetDevData`, on Star@.dev)

The wipe re-homes onto the plain `Star` (the dev data Star at `{u}.{g}.dev`); its **trigger** is the Flow 1b serverв†’client prompt (`nebula-dev-flows.md` Decision 11), not breaking-change detection.

- **`@mesh(requireAdmin)`, hard-guarded to the `.dev` STAR-tier instance** вЂ” segment-precise, **not** a suffix test: `const s = instanceName.split('.'); if (!(s.length === 3 && s[2] === 'dev')) throw` (matches the existing `#starBinding()` form at `nebula-client.ts:861`). `endsWith('.dev')` would also pass a galaxy-tier `acme.dev` вЂ” harmless in practice (a 2-segment name is a Galaxy, never routed here) but the precise form is free, so pin it. Body = `blockConcurrencyWhile(deleteAll() в†’ onStart())`.
- **Deliberate structuralв†’runtime weakening (owned, not overlooked).** Today the wipe lives *only* on the `DevStar` subclass **precisely so a tenant `Star` cannot carry a data-wiping reset** (`dev-star.ts`: "it must not exist there at all"). Decision 2's single-`Star` collapse ends that structural guarantee вЂ” the wipe now ships on **every** `Star`, gated only by the runtime `.dev` throw above. The compensating controls are the hard `.dev` guard + `@mesh(requireAdmin)` + the `@mesh`-surface-freeze test (below); the old `dev-star.ts` rationale is **superseded** because `DEV_STAR` no longer exists to isolate against. This move lands in **Phase 3.5**, before the `DevStar` class is deleted in Phase 4.
- `onStart()` must be a **complete re-init**: reconstruct DagTree / Resources / Subscriptions, recreate schema + `ROOT`, and **null `Star.#row`/`#facet`** so no stale validator-facet cache survives. Founder root-admin **reseeds** on the next admin call's `onBeforeCall` first-touch.
- **Caveats:** if `svc.alarms` is ever used, evict the cached Alarms service too; the bounded in-flight-write edge (generation-counter hardening) is deferred under the single-admin demo contract.
- **Data bargain:** additive ontology changes preserve data; breaking ones reset it (the user decides via the wipe prompt).
- **`@mesh`-surface-freeze must walk `Star.prototype`** (the post-collapse production surface), with `resetDevData` in its admin-gated set вЂ” mutation-validated by removing `requireAdmin` в†’ RED. вљ пёЏ The existing freeze (`dev-star-data-lifecycle.test.ts`) walks `DevStar.prototype` **own-properties only**; once `resetDevData` lands on `Star.prototype` that walk goes **vacuous** for it (dropping `requireAdmin` would pass silently). The `nonAdminMeshMethods(DevStar)` freeze is **subsumed by the `Star.prototype` walk, not inherited**.
- **Test the relocation вЂ” rewrite, don't drop.** The existing structural containment test asserts `Star.prototype.resetDevData === undefined` ("footgun guard", `dev-star-data-lifecycle.test.ts:365`); the move reds it. **Rewrite** it into the runtime-guard test: `resetDevData` on a **non-`.dev`** `Star` instance **throws + wipes nothing** (capable-of-failing by mutating the guard to always-pass), adding the instance-name operand alongside the existing admin-gate (B1) test.
- **`dev`-slug reservation вЂ” server-side enforcement is future work; accepted risk for the demo.** The guard's premise (no tenant owns a real `{u}.{g}.dev` Star) is **enforced client-side only** today вЂ” `claimStar` (`nebula-auth-registry.ts`) accepts `dev` as a valid slug. Server-side enforcement belongs in the **future Star/tenant self-service signup flow**, **NOT** in `@lumenize/nebula-auth` (reserving an app-level slug in a lower-level package is a layering smell). **Accepted risk until then:** a tenant self-claiming `{u}.{g}.dev` would get an admin-gated, **self-scoped** wipe of its own data (no cross-tenant exposure). Tracked: `tasks/backlog.md`.
- **Out of scope:** URL-level branching, seeding a dev Star from prod (fork-to-test), concurrent `dev-<name>` sandboxes (`tasks/icebox/nebula-branches.md`).

### Publish (Flow 2) вЂ” *post-demo, imminent*

Publish is a fast DevStudio-orchestrated command (Studio UI в†’ DevStudio в†’ DevContainer в†’ Galaxy), not a file push. Per Flow 2/2b: **DevContainer runs `vite build` в†’ pushes the built bundle + assets (R2 asset set) to Galaxy**; Galaxy is the **published-only** immutable app-version registry; a deployed (prod) Star **lazy-pulls** on version mismatch (Flow 2b). One unified version: app-version = ontology version (Decision 9).

- **Versioning granularity shifts ontology-level в†’ app-level.** Galaxy's existing `appendOntologyVersion` / `getLatestOntologyVersion` are **upgraded (renamed + rescoped), not deleted** вЂ” at publish they become *app-version* methods (built bundle **+** compiled ontology, lock-step by construction). That upgrade is **this** (post-demo) work; the **demo dev loop never touches the Galaxy registry** (DevStudio compiles the validator and applies it straight to the `.dev` Star вЂ” Decision 9). Phase 4 tears down only the now-unused *dev-loop pull path*, leaving these two methods in place flagged for the upgrade (see Phase 4).

- **Security:** Galaxy publish/admin methods carry `@mesh(requireAdmin)` вЂ” `<id>.*` scope widening admits descendant non-admins, so re-validate every browser-supplied field server-side. **Test patterns:** maintain the B5 frozen non-admin allow-list; a genuinely-minted in-scope non-admin must be **rejected + write nothing**; assert publish atomicity via debug-sink markers.
- **Post-demo seams** (none demo-blocking вЂ” the demo *is* the dev loop; prod serve = Flow 2b, out of demo scope):
  - **Binary file-assets (`ArrayBuffer` resources) вЂ” *right after demo*.** Generated apps will want a custom logo / images / fonts almost immediately; store + serve binary assets (as `ArrayBuffer`-valued resources or in the asset set).
  - **Custom-domain routing вЂ” *down the road*.** Map a user-developer's own domain onto their app. We do **NOT** chase "clean URLs": the `bindingName` + `{u}.{g}.{s}` instance-name convention is baked into how mesh routes, and fighting it is a permanent footgun. The scoped URL stays the URL вЂ” users won't parse "star", but they recognize the three dot-segments (company В· product В· their org).
  - **Prod version catch-up вЂ” *right after demo*.** When a prod Star first sees a newer app-version, Flow 2b lazy-pulls that version's validator/ontology from Galaxy; static assets serve from the edge regardless, so this is just the brief catch-up window (the first version-bearing op waits on the pull). *(The earlier "`Star.onRequest` can't await в†’ loading-shell + reload" framing was old-in-DO-serve salvage and no longer applies вЂ” serving is static.)*

## Studio-generated artifacts

The engine generates two artifacts that must stay coherent (the UI references entity/field names and access patterns that match the current ontology exactly):

1. **Ontology** вЂ” a `.d.ts` file (TypeScript types + annotations; see `nebula-agentic-development-engine.md` for the `@title`/`@description`/`@inverse` conventions). DevStudio compiles it to a runtime validator (dev) / Galaxy compiles for prod; the Star never compiles. Resources are served by the platform's `Resources` engine, configured entirely by the ontology.
2. **UI** вЂ” `.vue` Single-File Components (`<script setup>`, TS) bound to the factory's reactive store (`store.resources.*` / `store.ui.*`). Components import `{ store, client }` from the scaffolded `nebula.ts`; NebulaClient stays out of component code. Patterns: `website/docs/nebula/coding-your-ui.md`.

**Source-of-truth = DevStudio's shell `Workspace`** (SQLite + R2) + local git (isomorphic-git). There is no separate VFS and (now) no "files-as-resources" model вЂ” the source tree lives in DevStudio; DevStudio pushes it to the DevContainer (`applyChanges`) and re-pushes on cold boot. The agentic loop reads/writes the tree locally against the Workspace (the LLM hot path).

### Bootstrap files (baked framework skeleton + a seeded app)

The **framework skeleton** (vite config, `index.html`, `main.ts`, the `nebula.ts` bootstrap, HMR wiring) is **baked into the container image** (same for every app, version-coupled to the container/vite). DevStudio **pushes the per-app source** (`App.vue`, components, ontology `.d.ts`); first-run seeds a minimal `App.vue` + starter ontology (`nebula-dev-flows.md` В§ *Open / to confirm* вЂ” "First-run scaffold (Flow 0 В· new app)"; there is no standalone Flow 0 heading yet). `nebula.ts` is the one bootstrap file the Studio LLM may extend (per-type conflict resolvers, first-run resource bootstrap):

```typescript
import { createNebulaClient } from '@lumenize/nebula/frontend';
export const { client, store, ready } = createNebulaClient({
  appVersion: __APP_VERSION__,    // injected by the serving layer
  authScope: __AUTH_SCOPE__,      // parent galaxy {u}.{g}
  activeScope: __ACTIVE_SCOPE__,  // {u}.{g}.dev in dev; the deployed star in prod
});
try { await ready; } catch { window.location.assign('/login'); }
```

`appVersion`/`authScope`/`activeScope` are **injected by the serving layer** (DevContainer in dev; the static-serve in prod) вЂ” server-derived, never request-supplied (the scope-injection choke point above).

**Why baked вЂ” and the planned relaxation.** The long pole in container cold-start is **npm unpacking**; baking the approved **Vue + DaisyUI + Lucide** set into the image removes it entirely for those apps (together with the no-persistent-volume fact from *DevContainer dev loop* вЂ” there's nowhere to cache an install across instances). This is a **short-term optimization**: Cloudflare is adding **Container snapshots** (tracked in `nebula-dev-flows.md` В§ *Future capabilities*). Once snapshots land, we tolerate **one long `npm install` on first build** (the snapshot captures the result) and **open up the dependency surface** вЂ” dev-users can pull in more client-side libraries and version their dependencies at will. The invariant that never relaxes: the Studio system prompt always forces server-side **Resources** (no dev-user-authored server logic outside that surface).

### Same-origin requirement (verification checkpoint)

Generated apps + the Studio UI must serve all scripts **same-origin under strict `script-src 'self'`** (no CDN / `unsafe-eval` / inline import map). Under the canonical model this is satisfied **for free** by Workers Assets (Studio UI, Decision 3) + vite static serving (Preview app) вЂ” so it's a *verification checkpoint*, not a build task. Styling/icons (DaisyUI, Lucide [ISC вЂ” correct `ATTRIBUTIONS.md`]) are ordinary baked container deps. Prod = `vite build` в†’ Workers Assets (no running container; ~28Г— CSS win vs the retired in-DO bundle).

## Authoring environment вЂ” chat-first + live preview

**Web, chat-first.** ~90% of time is in chat; there is **no editable code editor and no file tree** outside the chat window вЂ” and **no separate pinned-ontology pane** either. When the engine shows code (incl. the ontology), it's **read-only within the chat**, surfaced inline and asked about. The recourse for "wrong code" is to talk to the engine, not edit it (a user edit desyncs the LLM's mental model). No "drop to the editor" escape hatch by default.

- **Persistent split:** chat one side, **live preview** (the feedback signal) always visible on the other.
- **Preview mechanism:** in-window **iframe** (same-origin per Decision 3 вЂ” auth via JWT over `/gateway`, shell via `/dev-container`); fallback = seamlessly **launch (or focus) a separate browser window** on the preview URL (window-handle reuse). **Auto-update is HMR** (vite `js-update`, patched in place ~milliseconds вЂ” not a full reload).
- **Preview-element highlight вЂ” *post-demo; the planned way to work*:** (the native gesture): the user clicks an element in the running preview; the engine maps the click to the source range; the user describes the change. They think in the preview, not in files. A text-span highlight in a chat-surfaced file is the precision fallback. Feasible because the Preview app is a **same-origin iframe** in Studio UI вЂ” Studio UI can read its DOM/state directly to do the clickв†’source mapping. (Shares the "observable preview" machinery with the engine's rendered-output self-feedback вЂ” see *Iteration loop*.)
- **Speech-to-text correction lane вЂ” *post-demo; the planned way to work*:** show the transcript in an editable field before send; optional auto-send after N seconds of silence with cancel + edit.

## Iteration loop

The engine must see what's broken to fix it.

> **The codegen loop's tool surface, error source, and bound are owned by [`nebula-codegen-loop.md`](archive/nebula-codegen-loop.md).** The **first loop** self-corrects from the **container-free Rung-1 compile** error-tail (SFC + ontology compile); the *runtime*-error path (`get_recent_errors`) below is **deferred there** (D8) until a Rung-2 runtime/preview signal is added.

- **Runtime-error feedback вЂ” *deferred (Rung-2; see [`nebula-codegen-loop.md`](archive/nebula-codegen-loop.md))*:** beyond compile errors, the loop will also self-correct from *runtime* failures вЂ” the kimi-ui-gen viability run wanted this (it caught an invented `op:'set'` that the *standalone SFC compile* can't, though a type-checked Rung-1 gate may; see the new file). **Then:** the engine reads recent runtime/validation failures via `get_recent_errors`. **Post-demo (rich):** extend `@lumenize/debug` to tail into the chat, namespace-scoped, with the engine subscribing to *suspect* namespaces so it focuses instead of drowning. (Compile errors are the first-loop channel вЂ” see the new file.)
- **Rendered-output self-feedback вЂ” *post-demo (big lift)*:** beyond runtime errors, let the engine observe the *running preview itself* and self-correct вЂ” the user says "element A floats over B" (or the engine notices unprompted) and it keeps adjusting CSS/HTML until fixed, autonomously. Two tiers: **structural** (DOM + computed styles + state) is cheap-ish вЂ” Studio UI reads the **same-origin Preview iframe** and serializes it to the engine; **visual** (pixel-level overlap/layout) is the harder tier вЂ” screenshot в†’ vision model. Increasingly tractable (headless-browser-driven-by-AI is now common; Claude Code ships preview self-feedback). Shares the "observable preview" machinery with the preview-element-highlight gesture (*Authoring environment*).
- **Agentic surface** (the tools the engine calls):
  - **Edit source** вЂ” `writeSource(path, content)`: the engine's core action. DevStudio commits it, then pushes the change to the DevContainer в†’ vite HMR (Flow 1). The **ontology `.d.ts` is edited exactly the same way** вЂ” it's just another source file.
  - **Read source** вЂ” the engine reads from DevStudio's Workspace directly (relevant files + the **pinned ontology** ride in context; on-demand reads as needed). No `get_current_ontology` вЂ” redundant now that the ontology is a source file, not a Galaxy-fetched registry entry.
  - **Runtime errors** вЂ” `get_recent_errors` (**deferred to Rung-2 вЂ” see [`nebula-codegen-loop.md`](archive/nebula-codegen-loop.md)**); `subscribe_debug_namespace` / `unsubscribe_debug_namespace` (post-demo). See *Runtime-error feedback* above for the why.
  - **Publish** *(post-demo вЂ” imminent)* вЂ” `publish()` (Flow 2).
  - *(Migrations are future + fuzzy вЂ” no concrete tool surface; see* Future capabilities *В§ Lazy schema migration.)*

## Conversation flow

Cold-start interview is the wow moment.

- **Interview pattern:** start with the **core entity** в†’ relationships в†’ workflows в†’ access patterns; reflect back; generate a **draft ontology early** as a live, correctable artifact.
- **Tone:** thoughtful product manager, not a form вЂ” make + flag assumptions, don't ask about every detail.
- **Wizard-style flow (the phases):** **context first** вЂ” understand *who* the user-developer is so the wizard tailors its detail + tone, then the product **vision & scale** (single-person / small-team / enterprise / multi-tenant), which surfaces the **access-control model** early. The Studio LLM can lean on [`website/docs/nebula/access-control.md`](../website/docs/nebula/access-control.md) вЂ” *the same app implemented two different ways* вЂ” to propose the right model for the situation. Then the **ontology** (the data-model interview above, validated coherent) в†’ the **UI** (against the validated ontology); back-and-forth throughout. (A migration-validation gate is post-demo; the demo wipes the dev Star on breaking edits via the Flow 1b prompt.)

## Future capabilities (tracked, not demo)

**Standalone** future directions (whole capabilities with no single home). Feature-specific post-demo items live *inline* with what they extend вЂ” e.g. Publish В§ *Post-demo seams*, *Iteration loop* (rendered-output self-feedback), *Authoring environment* (preview-element highlight) вЂ” and many smaller ones are in `backlog.md`.

- **Lazy schema migration** (the future of the breaking-edit wipe): a per-Star **copy-on-read** runner вЂ” runs against the routed Star's own SQLite with **no Star-identity awareness** (identical on `.dev` and prod; Decision 11 lockstep). Hard boundary: **default-fill is the parser's job, not migration's** (additive `@default` fills on next write; migrations are *only* for renames / type-changes / required-adds / computed). When built, it rides the **per-resource ontology-version stamp + `OntologyStaleError`**, never a Galaxy registry. **Many moving pieces, not one `propose_migration` tool (TBD):** the **non-deterministic** lazy-migration code is **proposed by the Studio LLM** and applied by that copy-on-read runner; separately, **deterministic SQL-schema migrations** only become necessary once user-developers can define their own **indexable fields**. Production-polish schema evolution (cross-resource callbacks, version skew, error UX) is the broader on-hold surface: `tasks/icebox/nebula-5.5-schema-evolution.md`. (Folded from `tasks/archive/nebula-lazy-schema-migrations.md`.)
- **Enterprise BYO-agent:** power users on their own coding agents (Claude Code / Cursor) expect a real **git + filesystem** surface, not the checkpoint metaphor. Worked example: GitHub-Enterprise + Claude-Code в†’ push-to-main webhook/GH-Action в†’ `git.clone` into a build DO's `Workspace` в†’ build в†’ push the built bundle to Galaxy (runs on `WorkspaceFileSystem` today, no Artifacts dep). Strongest argument for a post-demo desktop authoring shell. (Folded from `tasks/archive/nebula-file-storage-backend.md`.)
- **Built-artifact (compiled bundle) store** behind Galaxy's version API вЂ” keep the API backend-agnostic so the store choice (R2 asset set today / KV / shell `Workspace` / Artifacts) stays a `StateBackend` swap, not an API change. (If a SQLite-table backend is ever chosen, a PK change вЂ” e.g. adding `version` в†’ `(version, path)` вЂ” is a full table rebuild under a schema-version latch, not an `ALTER`.)
- **In-app AI chat** (every Nebula app auto-gets one): post-Studio RAG against the app's own data; same agentic substrate (`nebula-agentic-development-engine.md` В§ Two agentic contexts).
- **DevStudio Skills** (the open `SKILL.md` standard): `tasks/on-hold/nebula-skills.md`.

## Success criteria

**Automated test gates** (a green suite proves these):
- [ ] User-developer describes a data model in natural language в†’ working ontology + resources on the dev Star.
- [ ] Generated code reaches the DevContainer (`applyChanges`) and passes schema validation.
- [ ] **DevStudio source-of-truth round-trips:** `writeSource` persists to the Workspace + git commit (testable now); a DevContainer cold boot re-pushes the full tree and the preview comes back identical вЂ” **incl. the boot-race retry** (first `containerFetch` hits a not-ready container в†’ DO retries on `ContainerUnavailableError` в†’ eventually serves, Flow 1c) (the cold-boot half rides the `it.skip` e2e run with `wrangler dev`).
