## Goal
Two outcomes, sequenced so the milestone lands first and the breaking change is gated:
1. **Unblock the secret-less Claude-hosted CI lane (the token-drop)** — omit the Cloudflare-remote-binding test projects when no CF creds are present, via an **opt-out** flag. **Independent of the package work; lands first.**
2. **Extract `@lumenize/email`** (MIT, near-dependency-free transport **library** — its only dependency is the zero-dep sibling `@lumenize/debug`) and refactor `@lumenize/auth` + `@lumenize/nebula-auth` onto it, **collapsing the provider-by-subclass API into one provider-agnostic `AuthEmailSenderBase`** (provider chosen by env) — a **breaking but cleaner** change, sequenced behind a **coordination gate** so the rip-out doesn't collide with the concurrent mainstream session.

The package is *designed* to also back a mesh `this.svc.email` capability, **deferred** to its first consumer — see [Deferred](#deferred--later-pointers-not-built-here).

## Why this exists (context + the security story)
CI is green (memory `project_ci_cloud_tests`). The blocker for a secret-less hosted lane (Claude Code web / Dispatch) is plaintext + loggable env vars; the broad `CLOUDFLARE_API_TOKEN` is the scary one. Its **only** reason to exist in our secrets is the e2e remote-proxy — prod sends via the `EMAIL` binding directly (no token). `RESEND_API_KEY` + `TEST_TOKEN` are low-risk + scopeable → acceptable in plaintext; the CF token is not.

## Architecture — the breaking extraction
**Today**: `CloudflareEmailSender`/`ResendEmailSender` `extends AuthEmailSenderBase extends WorkerEntrypoint`; provider hardcoded per deployment by *which subclass* the `AuthEmailSender` entrypoint extends. `ResolvedEmail` **and** the 5-variant `EmailMessage` union both live in `packages/auth/src/types.ts`.

**Target**:
- **`@lumenize/email` (library) owns**: `ResolvedEmail` (moved from auth — a plain TS type, ADR-001), `EmailTransport` interface (`sendEmail(email: ResolvedEmail): Promise<void>`), `CloudflareEmailTransport` (reads `env.EMAIL`), `ResendEmailTransport` (reads `env.RESEND_API_KEY` + `fetch`), `createEmailTransport(env: object, { provider? })`. Plain classes (NOT `WorkerEntrypoint`); `env` typed `object` with narrowed optional access (`(env as { EMAIL?; RESEND_API_KEY?: string; EMAIL_PROVIDER?: string })`), not `as any`. No `cloudflare:workers` import; depends only on `@lumenize/debug` (which hides its own `cloudflare:workers` behind its `workerd` export condition).
- **`EmailMessage` (the 5-variant template union) STAYS in `@lumenize/auth`** — it's template/auth-domain, not transport. **Only `ResolvedEmail` moves.**
- **`@lumenize/auth` (breaking)**: ONE concrete **`AuthEmailSenderBase`** (`WorkerEntrypoint`) — keeps `EmailMessage` + the 5 templates + the `send(EmailMessage)` dispatcher; its `sendEmail()` **composes `createEmailTransport(this.env)`**. User-developers `class X extends AuthEmailSenderBase { from = … }`; **provider chosen by env.** `CloudflareEmailSender` / `ResendEmailSender` are **removed**.
- **`@lumenize/nebula-auth`**: `NebulaEmailSender extends AuthEmailSenderBase` (was `CloudflareEmailSender`); keeps `from`/`appName` + the `X-Lumenize-Auth-Instance` header override.

### Phase 5 — Quiet nebula-auth's fire-and-forget email-send rejections (cleanup; do last)
**Goal**: a `nebula-auth` run stops emitting the ~25 `uncaught (in promise)` "EMAIL binding is not configured" rejections. **Pre-existing noise** (the old `CloudflareEmailSender` threw identically on the no-binding path; `failed: 0` — Phase 3 only changed the message via `@lumenize/email`), so this is cleanup, not a regression fix. *(Moved here from `backlog.md` per Larry 2026-06-26 — a phase here won't get lost.)*
**Root cause (confirmed by run):** with no `EMAIL` binding but `RESEND_API_KEY` present (symlinked `.dev.vars`), `createEmailTransport` fell back to **real Resend API calls** that failed ("domain not verified", rate-limited). `AuthEmailSenderBase.send`/`sendEmail` both `await`, but the rejection escapes **entrypoint-side through the RPC boundary** (the DO's `await sender.send()` resolves before the rejection settles), so the caller's `try/catch` can't catch it — hence "Uncaught (in promise)".
**Success criteria**:
- [x] nebula-auth's fire-and-forget email sends are **handled** — chose the capturing-sender option: `test/test-worker-and-dos.ts` now exports a `NebulaEmailSender` **test subclass** overriding `sendEmail` to capture-and-drop (debug marker `nebula-auth.test.email`, no real provider call). Test-scoped, no production code change.
- [x] A `nebula-auth` run shows **no email-send uncaught rejections** — 0 email-related lines; 308 passed; type-clean. The 25 remaining unhandled errors are **all `RegistryError`** — unrelated baseline noise from the parent-child session's registry/claim work (`nebula-auth-registry`), not email.
- [x] Coordinated with `backlog.md`'s *Nebula email sender domain* "surface email-send failures" point — the fix is **test-scoped only**, so production failure-surfacing stays owned by that backlog item (no double-fix).
