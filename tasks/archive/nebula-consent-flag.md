## Objective

- **What:** add a **nullable** `improveProductConsent` column to the `NebulaAuthRegistry` `Instances` table, applied via the `@lumenize/sql-migrations` runner. **Consent is per-scope; Universe-level is the only level implemented now:** `claimUniverse` sets the Universe row to `1` (the slug-pick notice is the human decision point); sub-instance rows (galaxy/star) are left **`NULL` = "no level-specific decision yet → inherit from the nearest ancestor that has one"** (the Universe today). Corpus pool = `SELECT instanceName FROM Instances WHERE improveProductConsent = 1 AND instanceName != ?` (bind `PLATFORM_INSTANCE_NAME`; NULL sub-rows fall out naturally, the reserved `nebula-platform` row is excluded). **Assume `true` for now** (hard yes for F&F testers — no opt-out UI yet). The per-row column is already **per-level-ready**, so galaxy/star consent later is UI-only, never a schema change.
- **Why:** the **migration-critical** piece is locking the column shape in **before more instances claim** (one-way door — pre-alpha caveat) + writing consent at claim time. (The corpus filter is built now too but has no live consumer yet — the honesty constraint on that lives once in Notes/M6, not repeated here.)
- **The real reason it's a task file, not a quick edit:** this is **our first schema migration against a deployed prod DO** (`NebulaAuthRegistry` is live on `nebula.lumenize.com`). The `Instances` table already exists in prod, so editing the `CREATE TABLE` constant **silently no-ops** there (`IF NOT EXISTS` skips the existing table). The migration runner (the prereq package) is what makes the column add real, eager, and idempotent.

**Naming:** the column is **generically named** (`improveProductConsent` — consent to use your data to improve *the product*), never `nebula`/`studio`-specific, so `nebula-auth` stays product-agnostic. (Home rationale: self-signup is an auth feature; consent lives where the `claimUniverse` call happens — `packages/nebula-auth`.)

## Out of scope (explicit)

- **The migration runner itself** → [`sql-migrations.md`](sql-migrations.md). This task only *uses* `@lumenize/sql-migrations`; its vendoring, sync-API port, generic test suite, and docs are that package's task.
- **Opt-out UI + per-scope (galaxy/star) consent** — pre-alpha assumes `true` and implements **Universe-level only**. The model is already per-level-ready (`Instances` carries the column per-row; `NULL` = inherit), so adding galaxy/star consent later is **UI + an effective-consent walk-up resolution, NOT a schema migration**; the opt-out toggle (set a row to `0`) is likewise later. **Don't build the walk-up resolution now** — no consumer until turn-log inspection ships (YAGNI).
- **The `synthetic:true` flag / other `Instances` columns** — separate, YAGNI-deferred (pre-alpha caveat).

## What exists today (grounding — verified 2026-06-29)

- `NebulaAuthRegistry extends DurableObject` — a **raw DO**, not `LumenizeDO` (`workers-projects.md`: nebula-auth is raw-DO infrastructure). **No `onStart` hook** — the eager-at-boot equivalent is **synchronous setup in the constructor body** (`durable-objects.md`).
- Schema setup is **lazy today**: `#ensureSchema()` ([nebula-auth-registry.ts:47](../packages/nebula-auth/src/nebula-auth-registry.ts)) runs `for (const schema of REGISTRY_SCHEMAS) sql.exec(schema)` guarded by a `#schemaInitialized` instance flag, called at the **top of every public method**. It only runs `CREATE TABLE/INDEX IF NOT EXISTS` — **no column-evolution / ALTER path.**
- `REGISTRY_SCHEMAS` = `[Instances, Emails, Emails-index]` ([schemas.ts:123](../packages/nebula-auth/src/schemas.ts)). `Instances` = `(instanceName TEXT PRIMARY KEY, createdAt INTEGER NOT NULL) WITHOUT ROWID` ([schemas.ts:101](../packages/nebula-auth/src/schemas.ts)).
- `Instances` is INSERTed at **every** `INSERT … INTO Instances` site in `nebula-auth-registry.ts` — enumerate with `grep -n 'INSERT.*INTO Instances' packages/nebula-auth/src/nebula-auth-registry.ts` (5 at time of writing: `registerEmail` `INSERT OR IGNORE`, `claimUniverse`, `claimStar`, `createGalaxy`, `createStar`). **Only the `claimUniverse` site gains the consent write**; all others stay column-absent → `NULL` (inherit).
- The reserved platform pseudo-Universe `nebula-platform` (`PLATFORM_INSTANCE_NAME`, types.ts:139) is single-segment and `registerEmail`-creatable in prod — it must **not** be consented (excluded from backfill + corpus).
- The registry is **deployed** (`nebula.lumenize.com`); the migrations one-way door is open as of the 2026-06-26 first prod deploy.

## Decisions (pinned — consent/registry side; runner-mechanism decisions live in [`sql-migrations.md`](sql-migrations.md))

| # | Decision | Choice | Rationale |
|---|---|---|---|
| D2 | Where the migration runs | **Eager, in the registry's constructor body** (synchronous) — calls `@lumenize/sql-migrations` `runAll()`. **No `blockConcurrencyWhile`** (the migration has no `await`; `transactionSync` in the runner gives atomicity). **Replaces** the lazy per-method `#ensureSchema()` guard. | Registry is raw-DO (no `onStart`), so the constructor body is the home; wrapping fully-sync work in `blockConcurrencyWhile` is an unneeded rule-deviation. Add it only if a future migration step must `await`. |
| D4 | Baseline vs delta + column shape | **Model A — one mechanism (`REGISTRY_MIGRATIONS` is the single schema source of truth; `#ensureSchema` removed).** An **exported `REGISTRY_MIGRATIONS` constant** (not buried in the constructor, so it's independently testable — B1), **one single statement per id**: **id-1 = `CREATE TABLE IF NOT EXISTS Instances …`**, **id-2 = `CREATE TABLE IF NOT EXISTS Emails …`**, **id-3 = `CREATE INDEX IF NOT EXISTS idx_Emails_instanceName …`** — today's `REGISTRY_SCHEMAS` **verbatim** (same tables, same columns as prod — the *new* `improveProductConsent` column is added by **id-4 only**, never the baseline), **each statement its own id**, now **frozen as baseline migrations** (future schema changes are new ids, never edits to these). **id-4 = `ALTER TABLE Instances ADD COLUMN improveProductConsent INTEGER`** (nullable, no DEFAULT). **id-5 = backfill `UPDATE Instances SET improveProductConsent = 1 WHERE instanceName NOT LIKE '%.%' AND instanceName != ?`** with the runner's **`params: [PLATFORM_INSTANCE_NAME]`** (bound `?`, never interpolated). **id-4→id-5 cross-statement dependency** (id-5 references the column id-4 adds) is safe **only because `runAll()` runs the whole pending set in one `transactionSync`** where the ALTER is visible to the UPDATE — **never split these across separate `runAll()` calls.** | **Nullable on purpose:** three states are distinct (`NULL` = no level-specific decision/inherit, `0` = declined, `1` = consented — `coding-style.md` nullable exception); `NULL` keeps the model **per-level-ready**. A **fresh DO** runs id-1..5 (baseline created, column added, nothing to backfill); an **existing prod DO** (marker unset) runs *all*: id-1..3 `IF NOT EXISTS` no-op (tables exist), id-4 adds the column, id-5 backfills existing user Universe rows to `1`. **Both converge** — the load-bearing prod-path test (Phase 2). **Backfill invariant (M4):** `NOT LIKE '%.%'` matches every single-segment name; the **only** non-consentable single-segment row is `PLATFORM_INSTANCE_NAME` (`nebula-platform`, the sole reserved single-segment name — types.ts:139), excluded by `!= ?`. **If a second reserved single-segment name is ever added, both the backfill AND the corpus `!= ?` filters must widen — keep them a shared exclusion list.** (Tier = dot-count in `instanceName`; `parseId` knows it.) |
| D6 | Scope reminder | The prod `NebulaAuthRegistry` is a **singleton** — one live DO, but it **holds every instance record**, so correctness is critical and there's no second instance to catch a mistake. | Low fan-out, high stakes. The runner-level seed-old-schema test (Phase 2) is the only pre-deploy proof of the existing-DO path. |

### Phase 2 — Wire the runner into the registry + add the column
**Goal**: `NebulaAuthRegistry` runs `REGISTRY_MIGRATIONS` via `@lumenize/sql-migrations` eagerly in its constructor (D2/D4), adding the nullable column + the Universe backfill.

**Success Criteria**:
- [ ] The column is added **solely by the migration list — NOT** by also editing a `CREATE TABLE` constant. (Doing both double-adds it: a fresh DO would create the column then `ALTER ADD COLUMN` it again → `duplicate column name` on every cold boot.) `REGISTRY_MIGRATIONS` is an **exported constant** (**not buried in the constructor** so the prod-path test can invoke it directly — B1) implementing the **5-id shape pinned in D4** (id-1..3 = the baseline `CREATE`s verbatim — same columns as prod, **without** the new consent column; id-4 = nullable `ADD COLUMN`; id-5 = platform-excluded backfill with `params`). Exact SQL lives in D4 (single source); assert here only the **outcome** — after running, the column exists and `nebula-platform` is excluded.
- [ ] The registry constructor calls **`super(ctx, env)` first**, then runs the migration list **synchronously** (D2 — no `blockConcurrencyWhile`). The `#schemaInitialized` field **and every `#ensureSchema()` call site** (not just the method) are removed (N1).
- [ ] **Capable-of-failing test — the prod path (B1; the load-bearing one).** Because the migration runs in the constructor body, a registry stub has already migrated by the time a test holds it — so this test does **NOT** go through the registry constructor. **Harness — pin it (the existing nebula-auth tests reach the registry only by RPC stub, with no `ctx.storage` access, and `runInDurableObject` is used nowhere there):** register a **migration-free bare DO** (`class BareStorageDO extends DurableObject {}`, registered `new_sqlite_classes`) in `test/test-worker-and-dos.ts` + `test/wrangler.jsonc`, reached via `runInDurableObject` to get a **virgin `ctx.storage`**. There, hand-create the **old** column-less `Instances` schema (`instanceName, createdAt`), insert a **Universe**, a **galaxy (`u.g`)**, a **star (`u.g.s`)**, and a **`nebula-platform`** row, leave the kv marker unset, then run `new SQLSchemaMigrations({ doStorage: ctx.storage, migrations: REGISTRY_MIGRATIONS }).runAll()` (per the pinned API in [`archive/sql-migrations.md`](sql-migrations.md)) and assert: column added; **Universe → `1`**; **galaxy, star, AND `nebula-platform` → `NULL`**; marker at the highest id. **Re-run** (same instance) → no throw, no row changes. **And data-step idempotency across a cold construct (M8):** set the Universe row to `0`, then construct a **FRESH** `SQLSchemaMigrations` over the same storage (simulates eviction — `#lastApplied` resets, so it re-reads the *persisted* marker, not the in-memory cache) and `runAll()` → the `0` is **not** reset to `1`. Reddens on a bare `ALTER`, a `DEFAULT 1`, a missing platform-exclusion, or a re-runnable backfill.
- [ ] **Capable-of-failing test — the fresh path:** constructing the registry on a brand-new DO yields the (nullable) column with all `REGISTRY_MIGRATIONS` applied (marker at the highest id). Proves the constructor *wires* the runner. *(Idempotency-on-re-run is proven by B1's cold-construct M8 re-run + the runner's own run-twice tests — NOT by "re-constructing the registry," which a single isolate won't re-run, so don't assert that.)*
- [ ] `WITHOUT ROWID` write-cost discipline respected (`durable-objects.md` § write costs) — no new index for this flag (the corpus `SELECT` is a rare full scan, not a hot path; a partial index is YAGNI).

### Phase 3 — Record consent at `claimUniverse` + the corpus query
**Goal**: `claimUniverse` writes the Universe consent; verify the corpus selection.

**Success Criteria**:
- [ ] `claimUniverse` writes `improveProductConsent = 1` on its **plain INSERT** — `INSERT INTO Instances (instanceName, createdAt, improveProductConsent) VALUES (?, ?, 1)` (`slug` bound `?` per `security.md`/M5; `1` is the constant assume-true value). **NOT an `ON CONFLICT` upsert (revised at build 2026-06-29):** `claimUniverse` already throws **409 at `checkSlugAvailable`** if the row exists, and there is no `await` between that check and the INSERT, so the row provably does not exist here — the conflict branch is unreachable, and an unreachable `ON CONFLICT` would be dead, un-mutation-testable code. (A pre-registered-but-unclaimed *universe* row is anyway a non-scenario; universes are self-claimed.) Sub-instance INSERT sites (`claimStar`/`createGalaxy`/`createStar`/`registerEmail`) are **unchanged** — their rows stay `NULL` (inherit).
- [ ] **Capable-of-failing test — the consent write (normal path):** `claimUniverse('acme', …)` → `acme` appears in `listConsentedInstances()` (consent `1`). Mutation-check: drop the consent write from the INSERT → `acme` lands `NULL`, falls out of the `= 1` corpus filter → **this test reddens**.
- [ ] **Capable-of-failing test — the corpus filter:** corpus query = `SELECT instanceName FROM Instances WHERE improveProductConsent = 1 AND instanceName != ?` (bind `PLATFORM_INSTANCE_NAME`) returns a consented Universe; a sub-instance row (`NULL`), an explicit-`0` row, and the `nebula-platform` row are **all excluded**. The `0` row is a **test fixture** — no product path writes `0` until the deferred opt-out UI; it guards the filter `= 1` against weakening to `IS NOT NULL`. Keep the `NULL` and `0` excludes as **independent** assertions (mutate each separately) (M7).

## Notes / risks

- **First prod migration — the runner-level test + the deploy are the real proof.** Constructing the registry always runs the migration first (constructor body), so the *prod* (table-already-exists) path can only be exercised by the **runner-level** seed-old-schema test (Phase 2 — hand-seed the old table, invoke `REGISTRY_MIGRATIONS` directly, NOT via the registry constructor — B1) + the actual deploy. Treat that test as load-bearing, not optional.
- **Assume-true is opt-out by default** (consent presumed, no withdrawal UI yet) — a defensible pre-alpha F&F choice. Two honesty constraints keep it clean: **(read)** the corpus query ships now as a **guard test only**, no live consumer yet (turn-log inspection v0 / daily digest v1 are deferred); **(write)** consent is written from day one, but the Phase-4 notice — the consent moment for real users — **must render before the first non-Larry user is invited** (Wave-3). **Pre-invite, Larry is the only subject, so no real user's consent is recorded without a notice. Larry owns this sequencing and accepts responsibility (2026-06-29).** (This is why Phase 4 may trail the *build* but not the *invite*.)
- **Lifecycle reminder:** consent lives on the `Instances` row of the **Universe** because a Universe is 1:1 with its claiming user-developer (Universe admin) in pre-alpha, and `claimUniverse` is the human consent moment.
