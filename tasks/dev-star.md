## Goal

Studio needs an isolated place to build and test that is never production. That place is a **Star** addressed by a reserved 3rd-segment slug: `dev`. A user-developer's app is a Galaxy (`{u}.{g}`), so its sandbox is the Star `{u}.{g}.dev` — one per Galaxy. Production tenants live at `/{u}.{g}.{s}/...`; the dev sandbox lives at `/{u}.{g}.dev/...`.

It **inherits** everything a Star already has — its own SQLite, DAG tree, subscriptions, per-Star permissions, scope-isolation, and the Galaxy-admin access bypass — and adds a contained layer of **dev-only** behavior (eager version application, data reset, the Studio compile + reload pipeline), isolated in a `DevStar` subclass so a tenant Star never carries those capabilities. Most of the added behavior is shared code reused from the production path, not new — see *Implementation shape*.

## Reserved slug `dev`

Slug **format** is validated by `isValidSlug` in [parse-id.ts](packages/nebula-auth/src/parse-id.ts) (`/^[a-z0-9][a-z0-9-]*$/`, no leading/trailing/double hyphen) and `validateSlug` in [dag-ops.ts](apps/nebula/src/dag-ops.ts). `dev` is **format-valid but reserved**: a user-developer must not be able to create a *production* Star named `dev`.

- It's a **reservation, not a format rule** — `dev` passes `isValidSlug`; the platform just claims it. (Prior art for a reserved identity name: `PLATFORM_INSTANCE_NAME` / `isPlatformInstance` in [parse-id.ts](packages/nebula-auth/src/parse-id.ts).) *Where* it's enforced is open: Stars are created lazily on first touch ([star.ts:65](apps/nebula/src/star.ts) — "no explicit `createStar`"), so there's no slug-admission gate today and no tenant-provisioning flow either. For now this is a **don't-forget invariant**, not a built check — it lands wherever an explicit Star-creation / provisioning path is eventually designed. See Open considerations.
  - **Stronger under slug-derived binding** (§ *Naming & binding selection*): because the client routes a 3rd-segment-`dev` scope to `DEV_STAR`, `dev` is now *structurally* the sandbox binding, not just a name claim — a production tenant Star can never legitimately be named `dev` because its client traffic would route to `DEV_STAR`. This is **self-shadowing within one Galaxy only** (`matchAccess` still blocks foreign auds — not cross-tenant escalation), and it reinforces why the reservation matters; full enforcement still waits on a creation gate.
- **Future multi-sandbox** (post-demo): extend the reservation to a `dev-<name>` convention (`dev-feature-x`) — all legal under the regex, a flat 3rd-segment namespace. Not built for the demo; just don't paint it out.

## Access

The dev Star's users are the **Galaxy's admins** (the developers), not end-users. A Galaxy admin reaches it by switching active scope to `{u}.{g}.dev` (exact-match, passes `onBeforeCall` like any Star), and their `claims.access.admin` bypasses the per-Star permission checks inside — the same bypass that grants them access to any Star under their Galaxy. So **no dev-Star-specific seeding or grant is needed; every Galaxy admin can develop in it.** A non-admin has no access (no grant, no bypass) and is 403'd. **Correctness needs no special gating.**

**Decided (demo): no entrypoint guard.** Per-Star permissions already 403 a non-admin after the DO spins up, so a pre-spin-up guard buys only spin-up avoidance for unauthenticated traffic — a DoS/hygiene optimization, not a correctness requirement. Revisit only if dev-DO spin-up from unauthenticated traffic becomes a measured cost.

**Dev-only mutators are explicitly admin-gated.** The "403 a non-admin" reasoning above holds only for **resource ops**, which pass through the DAG `requirePermission` checks inside `resources.ts`/`dag-tree.ts`. The new dev methods (eager-apply, reset) are bespoke `@mesh` methods that do **not** go through those checks — and `onBeforeCall` proves only tenant *scope* (`matchAccess`), never `access.admin` ([nebula-do.ts:80](../apps/nebula/src/nebula-do.ts)). So each dev mutator carries **`@mesh(requireAdmin)`** directly (matching the `claims.access.admin` model above). Without it, anyone able to mint a valid `{u}.{g}.dev` aud could trigger a data-wiping reset or a forced re-apply.

## In-dev data lifecycle on ontology change

> **Single source of truth** for the dev-Star data-on-ontology-change bargain (additive-preserved / breaking-resets). `tasks/nebula-studio.md` and `tasks/on-hold/nebula-lazy-schema-migrations.md` point here — don't restate the rule there.

Studio's loop edits the ontology *within* the dev Star. The lazy in-place migration runner that would preserve data across those edits is **deferred** (`tasks/on-hold/nebula-lazy-schema-migrations.md`). For the demo we take the cruder bargain:

- **Additive edit** (new optional field with `@default`) → existing dev snapshots stay readable because reads return the stored value verbatim (no re-validation); the new field's `@default` materializes on the next *write*, not on read (mechanism detail + test in P3). No migration, no reset. Data preserved for free.
- **Breaking edit** (rename, type change, required-field add) → old snapshots no longer validate. We do **not** migrate them. Instead the dev Star is **reset to empty** and the user-developer rebuilds test data. This is acceptable because the demo narrative is "build from scratch," and breaking edits are most common early when the dev Star holds little.

Mechanism is a **full Star wipe via `ctx.storage.deleteAll()`** — it removes the entire private SQLite DB (SQL tables + KV + active alarm rows) in one call, so no **stored** data is stranded. Chosen over a surgical, enumerated wipe **precisely because** the surgical version silently leaks any newly-added table/key we forget to list; `deleteAll` is the less-brittle default. (The wipe is total; the *re-init* afterward is the part that must be kept complete — see the Lifecycle note for what `onStart` does and does **not** restore.) The DO instance and its `{u}.{g}.dev` registration survive — `deleteAll` clears *contents*, not the instance. This is a **wholesale sandbox teardown, deliberately outside ADR-004's scope** — ADR-004 ("history is the substrate; no destructive writes") governs the *resource write path* (an ordinary write never mutates or deletes a past snapshot), not the disposal of a throwaway dev sandbox. So the reset needs no ADR-004 exception; this paragraph is the explicit statement that it is out of scope. ⚠️ **It forecloses history-based restore**: after the wipe, Studio's named-checkpoint UX (`tasks/nebula-studio.md` § *Durable draft ownership*) cannot recover pre-reset state from dev-Star snapshot history. If checkpoints ship, they need their **own** copy mechanism (named snapshot / export), decided **jointly** with this reset — not built on the history a wipe destroys. Whether the reset is Studio-triggered on breaking-change detection or a manual "reset dev" action is pinned during the Studio build — see `tasks/nebula-studio.md`. The deferred runner replaces this reset with in-place migration when "your data evolves in place" becomes part of the product story.

> **⚠️ Reset precondition — UPDATED 2026-06-18, superseded by [`nebula-container-dev-loop.md`](nebula-container-dev-loop.md) Q#2: a DevStar wipe never touches source.** Under the container dev-loop, the user-developer's **source** lives in the **DevContainer DO's own storage** (re-hydrated into the container tree on cold boot), **not** the dev Star — the dev Star now holds **only dev DATA** (Resources). So `deleteAll()` wipes data but **cannot lose source** (different DOs), and the old precondition ("confirm source durable on Galaxy before wipe; re-hydrate from Galaxy") is **satisfied by the architecture** — there is nothing source-related to confirm or re-hydrate at the DevStar wipe. `resetDevData()` is safe to wire whenever its DATA-reset trigger is ready. *(Historical: in the superseded in-DO model the dev Star held source `file` resources dual-written to Galaxy; that role moved to the DevContainer, and Galaxy now holds only the **published** app-version registry, not dev-draft source.)*

> **Lifecycle note (build-critical):** `deleteAll()` drops the *tables themselves* **and** leaves warm in-memory caches stale — so the reset must rebuild both. Schema + ROOT live in the helper *constructors* (`CREATE TABLE IF NOT EXISTS` + DagTree `#ensureRoot` — [dag-tree.ts:38-77](../apps/nebula/src/dag-tree.ts), [resources.ts:82](../apps/nebula/src/resources.ts)), invoked from `onStart()` (cold-start only). DagTree also caches a permission/tree map (`#_cached`/`#_view`, invalidated only on its *own* mutations — [dag-tree.ts:28,100](../apps/nebula/src/dag-tree.ts)), so a SQL-only re-create would keep the OLD `#dagTree` authorizing *wiped* grants from its stale cache. The reset therefore **re-invokes `this.onStart()`** — it reassigns `#dagTree`/`#resources`/`#subscriptions` to fresh instances (empty caches) **and** recreates schema + ROOT for free (the constructors are idempotent — **no separate init helper**). One gap `onStart` doesn't currently cover: `Star.#row`/`#facet` (cached ontology + validator facet — [star.ts:39-40](../apps/nebula/src/star.ts)) aren't reassigned by `onStart`, and `#ensureFacet` short-circuits on a populated `#row` ([star.ts:124](../apps/nebula/src/star.ts)) — so **`onStart` must additionally null `#row`/`#facet`** (a no-op on cold start where they're already null; it makes `onStart` a complete (re)init). Because these are Star privates, this lives inside `Star.onStart` and is reachable from `DevStar.resetDevData` via `this.onStart()` — private access keys off the *declaring* class, not the instance. The founder root-admin grant is **not** re-seeded by `onStart` (only `onBeforeCall:86` seeds it); it **rides the next admin call's `onBeforeCall` first-touch** (`deleteAll` wiped the `__nebula_rootAdminSeeded` latch — [star.ts:82](../apps/nebula/src/star.ts)), so ROOT is grantless to the request-access climb for the brief window until that next admin call (acceptable — dev users are galaxy admins who reach it via the `claims.access.admin` bypass regardless, and the triggering admin's own next call reseeds). See P3 for the method shape + the honest test.
>
> **What `this.onStart()` does NOT restore (scope the claim):** it re-inits *helper-owned* schema + caches (DagTree/Resources/Subscriptions/TreeSubscriptions + `#row`/`#facet`), **not** base-constructor or lazy-service state. The `__lmz_alarms` table is created in the **Alarms service constructor** (lazy, cached in `LumenizeDO.#serviceCache`), and the base-constructor's overdue-alarm recovery isn't re-run by `this.onStart()`. **Latent today** — `apps/nebula/src` uses no `svc.alarms`, so this is inert — but if an alarm ever lands on a DevStar (a future Studio timer, `@lumenize/fetch`), `deleteAll` drops `__lmz_alarms` while the cached service survives → next `schedule()` throws `no such table` until eviction. Mitigation when that day comes: `resetDevData` also evicts the cached Alarms service so the next access recreates the table. So "nothing stranded" is true for *stored* data, not for this cached/lazy state.
>
> **Concurrency assumption (in-flight write vs reset):** `blockConcurrencyWhile` keeps the reset's own body atomic, but it does **not** abort a *concurrent* `doTransaction` already suspended at its `facet.parseBatch` await (that await opens the input gate — [resources.ts:365](../apps/nebula/src/resources.ts), see the interleave comment at :416). Such a write resumes **after** the wipe and commits a Snapshot into the freshly-emptied tables — re-populating the sandbox and possibly orphaning `Snapshot.nodeId → Nodes`. For the demo this is a **bounded edge**: the dev sandbox is single-admin and Studio's loop quiesces its in-flight write queue before triggering reset (it's a disposable sandbox the user is rebuilding anyway). Documented + characterized by P3's negative test; the server-side hardening (a generation counter checked after `parseBatch`, before `transactionSync`, aborting writes whose generation predates the last reset) is **deferred** — it would touch the shared production transaction path.

## Implementation shape

- **`DevStar extends Star`.** The dev-only behavior — eager version application, the breaking-edit data reset, the Studio compile + reload-broadcast pipeline — lives on a `DevStar` subclass, **not** behind a runtime flag on `Star`. Keeping it off the base class means a tenant Star *can't* carry these capabilities; a data-wiping reset reachable on a tenant Star would be a serious footgun, so it must not exist there at all. Instance name, scope-isolation `onBeforeCall`, and per-Star storage are all inherited unchanged — `DevStar`'s bare 3-tuple `{u}.{g}.dev` flows through Fix 1's `buildAuthScopePattern → matchAccess` exactly like any `Star`, so **no scope-isolation-side wiring is needed** (the binding-aware grammar that would have required registering `DEV_STAR` belonged to the deferred Fix 2). If the subclass can't reach something it needs, add a clean extension point (hook) to `Star` rather than inlining dev logic into the base.
- **Naming & binding selection.** Class `DevStar`; DO binding `DEV_STAR` (SCREAMING_SNAKE, matching `STAR`/`GALAXY`/`UNIVERSE`). **The binding is selected from the active-scope slug**: a caller addressing `{u}.{g}.dev` targets `DEV_STAR`; any other star slug targets `STAR`. This is the one safe design, because of a sharp footgun: the Gateway **trusts the client-supplied binding** (`#handleClientCall` → `getDOStub(env[binding], instance)`, no re-derivation — [lumenize-client-gateway.ts:586](../packages/mesh/src/lumenize-client-gateway.ts#L586)), and `env['STAR'].idFromName('{u}.{g}.dev')` vs `env['DEV_STAR'].idFromName('{u}.{g}.dev')` are **different DO instances with separate SQLite** that `onBeforeCall` *both* accept (the scope check is name-derived, binding-agnostic). So a wrong/forgotten binding does **not** error — it silently reads/writes an empty wrong-namespace Star. Deriving the binding from the slug at a **single choke point** is therefore both necessary and sufficient. **This choke point is a *correctness* control, not the isolation boundary** — tenant isolation rests on the inherited `onBeforeCall` name-derived scope check (which rejects a foreign `aud` regardless of which binding the client named), *not* on the client picking the right binding. The worst a malicious client achieves by pairing `binding='STAR'` with `instance='{u}.{g}.dev'` is reaching a *different, empty* Star at the same name whose `onBeforeCall` enforces the same `aud` — never another tenant's data.
  - **Client (the resource hot path).** `NebulaClient` selects the binding from `#activeScope` via one helper applied at **every** Star-targeting call (the ~16 `lmz.call('STAR', …)` / `callRaw('STAR', …)` sites in [nebula-client.ts](../apps/nebula/src/nebula-client.ts), incl. the reconnect `subscribeTree`). The helper is a **binary choice between two literal binding-name constants** — `#starBinding() { return this.#activeScope.split('.')[2] === 'dev' ? 'DEV_STAR' : 'STAR' }` — **not** a case-conversion or `env` lookup (that's the routing-layer smart-match below, a *different* mechanism). There are exactly two possible targets, both known at authoring time, so the slug only picks *which* literal; it is never *converted into* a binding name. Use the **local slug check**, **not** an imported `parseId` — `nebula-client.ts` is browser-bundled and must stay free of `cloudflare:workers` (packaging.md), and a value import of `@lumenize/nebula-auth` would drag the barrel's DO classes (and `cloudflare:workers`) into the bundle (no browser-safe subpath exists today). `#activeScope` is set at construction and on every scope-switch, so the slug is always known at call time.
  - **DO→DO callers** (e.g. Galaxy → dev Star) name `DEV_STAR` literally: `lmz.call('DEV_STAR', '{u}.{g}.dev', …)`. Both routes agree because both map slug `dev` → `DEV_STAR`.
  - **HTTP / `routeDORequest` is NOT on the resource flow** — resources go client → Gateway (WS) → `env[binding]`; there is no direct HTTP-to-Star path today. If one is ever added, `getDONamespaceFromPathSegment` smart-matches on the **binding name**, generating case-variations of the kebab segment `dev-star` → `{DEV_STAR, DevStar, devStar, …}`. ⚠️ Expose exactly **one** env binding *name* that smart-matches `dev-star` (use `DEV_STAR`); never add a second DO binding whose **name** is also a case-variation (e.g. a `DevStar` binding) → `MultipleBindingsFoundError`. (The *class* may be named `DevStar`/`DevStarTest` — class names are irrelevant to the smart-match, which keys on `Object.keys(env)`.)
  - **Connection binding — resolved (no DevStar work):** the WS connection targets `NEBULA_CLIENT_GATEWAY` (`gatewayBindingName`); each call carries its own target binding; push keys on `callee.instanceName` and subscribe captures `subscriberBinding` per-row from `callChain.at(-1)`. The connection `attachment.bindingName` is the gateway/client binding, never a Star-target binding. So DevStar needs no connection-time change — the per-call `#starBinding()` choke point is the whole story.
- **Eager version application, reusing the existing apply path.** A production tenant Star picks up a new ontology/app-bundle version *lazily*: on a read cache-miss it fires the cross-DO continuation `lmz.call('GALAXY', this.#galaxyId, ctn<Galaxy>().getLatestOntologyVersion(), ctn().<handler>($result))`, whose handler applies the fetched `OntologyState` via `Star.#installState` ([star.ts:162](../apps/nebula/src/star.ts#L162)); `compileOntologyVersion` is already an exported pure fn ([galaxy.ts:66](../apps/nebula/src/galaxy.ts#L66)). **The apply is a mesh round-trip, not a synchronous call** — `#installState` needs an `OntologyState` in hand, and `getLatestOntologyVersion` is a `@mesh()` method on Galaxy ([galaxy.ts:133](../apps/nebula/src/galaxy.ts#L133)), unreachable synchronously. For dev-cycle speed the dev Star runs that **same** path *eagerly* on `deploy_to_dev` instead of waiting for a cache-miss — the **eager analog of the lazy Handler-1**: `lmz.call('GALAXY', this.#galaxyId, ctn<Galaxy>().getLatestOntologyVersion(), this.ctn().applyFetchedState(this.ctn().$result))`. Eager-vs-lazy is just *when* the existing path runs — **no new shared "apply module," and no abstraction for the deferred tenant lazy-migration runner** (`tasks/on-hold/nebula-lazy-schema-migrations.md`, which doesn't exist yet). The one real extraction: `#installState` is `#`-private, so the continuation handler can't be the subclass — add a narrow **`protected applyFetchedState(state)` hook** on `Star` (wraps `#installState` on a non-null/non-Error state) that both the lazy handlers and the eager trigger call. No dev-only branch in the base. `applyFetchedState` is **safe-by-construction** the same way `doTransaction`/`doRead`/`doSubscribe` are — `protected`, **not** `@mesh`, so it resolves only as an internal continuation handler (via the local executor), never client-reachable; the `@mesh(requireAdmin)` gate lives on the *trigger*, and the hook itself needs no guard. **Null-state contract:** `getLatestOntologyVersion()` returns `null` on an un-published Galaxy ([galaxy.ts:135](../apps/nebula/src/galaxy.ts#L135)), so `applyFetchedState(null)` is a deliberate **no-op** — `deploy_to_dev` always publishes the version to Galaxy *before* triggering eager-apply, so a null means "nothing to apply," not an error.
