# Star root-admin bootstrap + last-admin protection

**Status**: Part 1 DONE 2026-06-14 (nebula-frontend v3 Phase 0). Part 2 (last-admin protection) REVIEWED & ready-to-build 2026-06-15 (/review-task panel). **Model context (clarified 2026-06-10).** There is no separate "Star admin" scope concept: a Star admin is simply a user holding `admin` on the Star's **root orgTree node** (`ROOT_NODE_ID`). A new tenant self-signs-up via the Galaxy (create account → pick Star name + slug → Galaxy provisions the Star), and the founder must come out of that flow holding `admin` on root. Every other user starts with **zero** grants and acquires access via request-access (climb to the nearest admin ancestor) or an app-developer invite — there is no auto-created per-user "home node". The root admin grant is therefore the single guaranteed starting point that makes the whole permission graph reachable: the request-access climb terminates at root only because the founder's admin grant lives *in the tree* there (a scope-level bypass admin wouldn't be findable by the climb).

This is the prerequisite the docs' founder-demo bootstrap leans on (coding-your-ui § Lists attaches the founder's `todoList` under `ROOT_NODE_ID`, valid precisely because the founder holds `admin` there).

> Resolved (deep-review flag #5, 2026-06-10/11): the `claims.access.admin` bypass at [dag-tree.ts:158](../apps/nebula/src/dag-tree.ts:158) is **kept** — it IS the Galaxy/Universe scope-admin tier, parallel to DAG grants and deliberately absent from the permissions map. The code comment is already relabeled ("Galaxy/Universe-scope admin bypass (NOT a Star admin — that's a DAG `admin` grant on root)"), and the two-source admin model is pinned in nebula-frontend.md § Decisions pinned, resources.md § Access control, and coding-your-ui § Gating admin-only UI. ## Part 1 — Seed the founder as root admin at Star provisioning — **DONE 2026-06-14** (nebula-frontend v3 Phase 0)

When the Galaxy provisions a Star for a new tenant, the founder's `sub` gets `admin` on `ROOT_NODE_ID` of that Star's orgTree: `dagTree.setPermission(ROOT_NODE_ID, founderSub, 'admin')`, run as part of the provisioning path (Stars are currently lazy DO instances — the seed happens on the first-provision flow, wherever the self-signup lands it; **no explicit `createStar` mesh method exists yet** — `apps/nebula/src/galaxy.ts` has config/ontology methods only).

**Chosen entry point (lazy first-touch): `Star.onBeforeCall()`** ([apps/nebula/src/star.ts](../apps/nebula/src/star.ts)). On the first authenticated touch, if the caller is the scope-admin founder (`claims.access.admin`, minted iff `subject.isAdmin` — nebula-auth.ts:1359), seed `setPermission(ROOT_NODE_ID, callerSub, 'admin')`, guarded exactly-once by a `__nebula_rootAdminSeeded` KV flag. No un-guarded path needed: `setPermission`'s own `admin` gate is satisfied by the scope-admin bypass in `requirePermission` (dag-tree.ts:158). A non-admin first caller leaves root adminless until an admin connects.

## Part 1b — Place the Star near the tenant (locationHint) — **DEFERRED 2026-06-15 (premise stale; blocked on mesh-origin-request)**

**Why deferred (design-review 2026-06-15).** This part assumed "Galaxy-provisions-Star places every Star near the Galaxy," but Part 1 shipped lazy first-touch in `Star.onBeforeCall()` — the founder's first authenticated call routes **client → Gateway → Star directly** ([star.ts:80](../apps/nebula/src/star.ts)), so the **Gateway** creates and pins the Star *eyeball-local* on first touch. The placement bug this part set out to fix doesn't occur in the shipped design, and the two source-paths originally sketched here are both dead ends:
- *Derive in place* (`cfToLocationHint(callContext.originRequest?.cf)`): any in-Star code, incl. `onBeforeCall`, runs **after** placement is already pinned — too late.
- *Persist on the Registry at the claim step*: `claimStar` never touches the Star DO, and the Gateway (the actual first-toucher) never consults the Registry — a write with no reader.

The only mechanism that could actually place the Star is **the Gateway computing `cfToLocationHint(attachment.originRequest?.cf)` and threading it through an options-passing `getDOStub`/`getByName(name, options)`** — exactly the "Gateway-derived hint on every resolution" item that [mesh-origin-request.md](mesh-origin-request.md) **explicitly defers as non-v1**. None of `CallOptions.locationHint`, `originRequest`, or `cfToLocationHint` exist yet.

**Revisit when** the self-signup flow gains a real pre-create Star-provisioning entry point (the [star.ts:76](../apps/nebula/src/star.ts) `TODO(self-signup)`) **and** `mesh-origin-request.md` un-defers Gateway-derived-hint-on-resolution. At that point the placement decision must be made at the first `getByName` that **creates** the Star (never by in-Star code) — pin that ordering invariant there. Until both land, this part is **BLOCKED** — do not pick it up in `/build-task`.

## Part 2 — Last-admin protection (enforce in dag-tree) — **REVIEWED 2026-06-15, ready to build**

**Invariant: `ROOT_NODE_ID` must always retain at least one *in-tree* `admin` grant** — a row in the `Permissions` table. That grant is the only climb-findable admin terminus: the request-access climb reads the permissions map, while the `claims.access.admin` scope-admin bypass ([dag-tree.ts:158](../apps/nebula/src/dag-tree.ts)) is **not** in the map and so does **not** preserve reachability. Enforce in dag-tree (it lives on the Star) so it holds regardless of caller — add-before-remove: to hand off, grant the new admin *first*, then revoke the old. The count deliberately **excludes** the scope-admin tier, so even a Galaxy/Universe scope-admin must add-before-remove.

Guard the two mutation points ([dag-tree.ts](../apps/nebula/src/dag-tree.ts)):

- [ ] **`revokePermission(ROOT_NODE_ID, sub)`** — reject if `sub` is the **last** `admin` grant on root (count of root grants with tier `admin` would drop to 0). Idempotent no-op (sub has no grant) stays a no-op. **Ordering:** the last-admin check runs **after** `requirePermission(ROOT, 'admin')` and immediately before `transactionSync`, so a non-admin caller gets `PermissionDeniedError`, never a `LastRootAdminError` that would disclose invariant state. (Disclosure is harmless today because the org tree is universally visible — `addEdge` M7 caveat, [dag-tree.ts:203-227](../apps/nebula/src/dag-tree.ts) — but keep the one consistent rule for a future per-branch-visibility world.)
- [ ] **`setPermission(ROOT_NODE_ID, sub, level)`** — reject a **demotion** of the last admin: if `sub` currently holds `admin` on root, `level !== 'admin'`, and `sub` is the only root admin → reject (same typed error). The compound predicate short-circuits on `nodeId !== ROOT_NODE_ID` or `level === 'admin'`, so non-root grants and promotions (incl. Part 1's `onBeforeCall` seed) never reach the count.
- [ ] **Typed error — `LastRootAdminError`** in [errors.ts](../apps/nebula/src/errors.ts), following the `OntologyStaleError`/`PermissionDeniedError` template: `override name = 'LastRootAdminError'`, a discriminating own prop (`nodeId: ROOT_NODE_ID`), and an exported `isLastRootAdminError(err)` guard checking `err.name` + own-prop presence (**never** `instanceof` — it doesn't survive the mesh hop). Put the actionable text ("add another admin before removing this one") in the Error `message` so even an unmapped client shows something useful. Wire client-side detection where `nebula-client.ts` already consumes `isOntologyStaleError`, so the user-developer UI gets the guidance; specify where dagTree mutation errors surface to the client.
- [ ] `deleteNode(ROOT_NODE_ID)` is already forbidden (root cannot be deleted/undeleted/renamed — [api-reference.md § client.orgTree](../website/docs/nebula/api-reference.md#clientorgtree)); confirm-only, no new guard.
- [ ] Compute "last root admin" from the cached permissions view (`#cached.permissions.get(ROOT_NODE_ID)`), counting entries with tier `admin` — all in-memory, no extra reads. **ADR-005 note:** "is this the last admin" is *non-monotonic*, which ADR-005's corollary normally keeps *inside* the transaction; reading it from cache pre-`transactionSync` is sound here **only because** the mutators are synchronous with **no `await`** between the cache read and the transaction and the Star is single-threaded, so no concurrent grant change can interleave. State this so a `/build-task` verifier doesn't false-positive an ADR-005 violation.

**Known limitation — grant-count, not liveness (accepted 2026-06-15):** the guard protects the *count* of root admin grant rows, not that the admin `sub` is still a live, authenticatable subject. Subjects live in `nebula-auth` (a lower-level DO; no FK from `Permissions.sub`), and its subject-delete path ([nebula-auth.ts:582](../packages/nebula-auth/src/nebula-auth.ts) `#handleDeleteSubject`) can't — and must not — reach into the Star (nebula-auth depending on Nebula's permission model would invert the dependency direction). So deleting the subject holding the last root-admin grant leaves a *dangling* grant: the invariant reads "held" but the org may be effectively adminless. **Why it's acceptable:** (1) a deleted subject can't authenticate, so the grant is unusable, not exploitable; (2) self-delete and bootstrap-admin delete are already refused at nebula-auth ([:586](../packages/nebula-auth/src/nebula-auth.ts)/[:598](../packages/nebula-auth/src/nebula-auth.ts)), which blocks the only *unrecoverable* case; (3) **recovery** is always available — a Galaxy/Universe scope-admin (`claims.access.admin`) re-seeds a live root admin via `setPermission(ROOT, liveSub, 'admin')` through the bypass. Support-ticket fix if it ever surfaces. The **user-facing fix** — detect dead admins and route around them at request-access climb termination (async-native there, so it doesn't disturb this synchronous grant-count guard) — is captured in [nebula-request-access.md](nebula-request-access.md); a broader cross-DO data-consistency sweep is backlogged ([backlog.md](backlog.md)). A revoke-time liveness check was considered and rejected (it would break the synchronous-mutator invariant above) — see that file.

**Tests** (each its own `uniqueStar()`; multi-step sequences run as ordered awaits within a SINGLE `it` on one Star — never split across `it` blocks):
- [ ] Single root admin → `revokePermission(ROOT, founderSub)` rejects with `LastRootAdminError`; the grant survives.
- [ ] Single root admin → `setPermission(ROOT, founderSub, 'write')` (self-demote) rejects; grant stays `admin`.
- [ ] **Operand — level:** `setPermission(ROOT, lastAdmin, 'admin')` (re-assert) does **not** reject (the `level === 'admin'` short-circuit).
- [ ] **Operand — only-admin:** with two root admins, `setPermission(ROOT, oneOfThem, 'write')` (multi-admin demote) **succeeds** — guards against a too-aggressive guard that blocks *all* root-admin demotions.
- [ ] **Operand — count:** add-before-remove: `setPermission(ROOT, second, 'admin')` then `revokePermission(ROOT, founder)` → both succeed, `second` remains admin; then `revokePermission(ROOT, second)` rejects (survivor is now the last).
- [ ] **Cross-boundary (ADR-002):** via `client.orgTree.revokePermission(ROOT, lastAdmin)`, the awaited mesh call rejects with an error whose `name === 'LastRootAdminError'` and discriminating own prop survive the Star→client hop, and `isLastRootAdminError` accepts it while a bare no-prop `Error` does not. (Mirror `nebula-client-ontology-stale.test.ts`.)
- [ ] Non-root nodes are unaffected — revoking/demoting the last admin on a non-root node is allowed (those can legitimately go adminless; access flows from ancestors).
- [ ] **No concurrency-race test** — mutators are synchronous and the DO single-threaded, so the count→mutate window cannot interleave and ADR-005's cross-hop concern doesn't apply; such a test would be vacuous. Recorded so a builder doesn't add a misleading green test.
