# Conflict / outcome state machine (v3 isolation detour)

**Status**: spec'd 2026-06-11, not started. Pre-v3 isolation detour for [nebula-frontend.md](nebula-frontend.md) Phase 5.3.7-v3 — extracted to its own file (like [debounce-serial-queue.md](debounce-serial-queue.md)) because it rivals debounce in invariant count and is far cheaper to validate against a mock-client harness than to debug once wired into the factory + Vue. **Load-bearing for the demo** (every transaction resolves through it) — pre-demo, built in isolation *before* the v3 wiring. Depends on: the submission queue ([debounce-serial-queue.md](debounce-serial-queue.md)) and [factory-textmerge.md](factory-textmerge.md) (for the `@longform` auto-resolver).

This is the engine that turns a server transaction response into (a) a top-level `TransactionOutcome` the `await` site sees, and (b) per-resource `TransactionResourceResolution`s delivered to the handlers, while driving optimistic-state effects (commit / rollback / repaint) and conflict-resolution chains. Types: [api-reference.md § TransactionOutcome / TransactionResourceResolution / ConflictResolverVerdict](../website/docs/nebula/api-reference.md#transactionoutcome).

## Invariants to validate in isolation

1. **Always-resolve, never-reject.** `transaction()` resolves with a `TransactionOutcome` for every terminal state; infrastructure throws (network/mesh) are wrapped as `{ kind: 'infrastructure-error', error }`. No code path rejects the Promise after this lands.
2. **Mixed-fate per resource.** Each resource in a batch resolves independently; the top-level outcome is `'ok'` whenever the server responded — even when some/all resources land at `'use-server'`, `'permission-denied'`, etc. `outcome.resources` carries each resource's final `TransactionResourceResolution`.
3. **Per-resource pre-write-snapshot rollback.** Capture a pre-write snapshot per `(rt, rid)` **at first divergence** — the value the store held before the first optimistic write of the burst, the SAME snapshot used as the merge `base` (invariant 5). NOT at submit time (the store already holds the user's edits by then → rollback would be a no-op "revert"); NOT one batch-wide `preWriteFullValue`. Rollback outcomes (`timeout`, `infrastructure-error` → roll back ALL; per-resource `validation-failed` / `permission-denied` / `retries-exhausted` → roll back that resource) restore from its own snapshot. `ontology-stale` does NOT roll back (page reload expected); neither does a mere disconnect — `timeout`/`infrastructure-error` are connection-gated (invariant 10).
4. **`use-this` recursion bounded by `maxRetries`.** `conflict-pending` → handler returns `{ kind: 'use-this', value }` → re-submit at the server's current eTag → maybe conflicts again → … → terminates at `'committed'` or `'retries-exhausted'` (after `maxRetries`, default 5).
5. **`base` threading (deep-review B4; capture-point corrected 2026-06-11).** Each `conflict-pending` carries `base` = the value the conflicting submission was based on. **Unifying rule: `base` always equals the value at the baseline eTag the next submission uses, and re-anchors whenever that baseline advances** — three sites: (a) **first divergence** = the last-confirmed value the store held *before* the first optimistic keystroke overwrote it (NOT the submit-time store value, which already equals `local`); (b) **commit boundary** = a cleanly-committed transaction's value when a buffered write chains onto it (`InFlightWithBuffer → InFlight`, see [debounce-serial-queue.md](debounce-serial-queue.md)); (c) **`use-this` re-conflict** = the previous conflict's `server` snapshot. `base` is never a server history lookup — it's client-held (the same snapshot kept for optimistic rollback), and it *cannot* be server-sourced: the same-actor in-place debounce overwrite destroys intermediate values, so a divergence point is frequently unrecoverable from history (verified 2026-06-11; ADR-004 corrected). See nebula-frontend.md's B4 record and api-reference § textMerge.
6. **Handler fall-through (deep-review M9).** A per-call override layers in front of — does not replace — the per-type handler, and is consulted for *every* resource in the batch. `conflict-pending`: per-call → (`undefined`) per-type → (`undefined`) framework default (`use-server`); first non-`undefined` verdict wins. Terminal: per-type fires for every resource; per-call layers additive; a per-call handler scoped by `resourceId` no-ops for the rest.
7. **Async resolver suspends the queue timeout.** While a `conflict-pending` handler awaits (modal), the 5–10 s in-flight timeout is suspended; a fresh timeout starts on each post-resolver re-submit. No max duration on the handler itself.
8. **Hold-pending-fanouts (mid-edit fanout contract).** While a resource has pending optimistic state (debounced write outstanding OR transaction in flight), incoming fanouts for it are **held**; the next submit's resolver sees both the full pre-fanout local state and the buffered server snapshot, merges, applies, releases. Resource-level granularity. Without it, `textMerge` only protects keystrokes typed *after* a fanout — partial and timing-dependent. This same hold covers the **reconnect initial-snapshot push**: on reconnect, auto-resubscribe delivers a fresh server snapshot; if the resource still has pending optimistic state (writes held through the disconnect, invariant 10), that snapshot is held too, then the flushed write commits-or-conflicts-and-merges — so pre-disconnect keystrokes survive the reconnect.
9. **Default flash classes.** `committed` → `lumenize-commit-success`; rollback outcomes (`use-server`, `validation-failed`, `permission-denied`, `retries-exhausted`) → `lumenize-conflict-revert`; `human-in-the-loop` / `conflict-pending` → none. Applied before the handler's terminal call so the handler can override.
10. **Connection-gated rollback (decided 2026-06-11).** The `timeout` and `infrastructure-error` timers are **suspended while `lmz.connection.state !== 'connected'`** — a mere disconnect NEVER rolls back optimistic state. A transaction in flight when the connection drops keeps its optimistic paint; on reconnect it re-submits with its *same* `newETag` (idempotent — the server short-circuits if it had committed). `timeout`/`infrastructure-error` fire (and roll back) only for failures that occur while *connected*. This makes the disconnected-banner promise honest and reserves rollback for server-acknowledged failures + genuine while-connected non-response. Pairs with the [debounce-serial-queue.md](debounce-serial-queue.md) connection gate (flush suspended while disconnected); the reconnect snapshot is held per invariant 8.

## Harness

Mock `client.transaction` (or the layer below it) that, per submitted batch, returns scripted per-resource results: `committed` / `conflict(serverSnapshot)` / `validation-failed(errors)` / `permission-denied` / and transaction-wide `timeout` / `infrastructure-error(throw)` / `ontology-stale`. Register per-type and per-call handlers. Assert: the resulting `TransactionOutcome` + `outcome.resources` map; each resource's optimistic-state effect (committed value / rolled-back-to-snapshot / painted-and-held); flash classes toggled; `use-this` re-submission count and the eTag/`base` each carried.

## Tests (state-machine / property style)

- [ ] Single resource: `committed` / `use-server` / `use-this`(1 retry→commit) / `use-this`(retries-exhausted) / `validation-failed` / `permission-denied` / `human-in-the-loop` — each yields the right resolution + optimistic effect + flash.
- [ ] **Mixed-fate batch:** `{ A: use-server, B: permission-denied, C: committed }` → top-level `ok`, `outcome.resources` has all three, each rolled-back/committed per its own resolution.
- [ ] **Handler fall-through (M9):** per-call handles only A; B's per-type set-union handler still runs on B's conflict (not shadowed → framework default).
- [ ] **`base` threading (B4) — all three re-anchor sites, each capable-of-failing:** (a) **first divergence** — capture base at the first optimistic write; conflict → base = the pre-typing value (a submit-time capture yields `base===local`, fails); (b) **commit boundary** — T1 commits cleanly, a buffered write becomes T2, a third party has moved the server → T2 conflicts; assert base = T1's committed value so the post-commit local edit AND the third party's edit both survive (a no-re-anchor impl reuses the original pre-write base and double-counts T1's edit, fails); (c) **`use-this` re-conflict** — force a re-conflict, assert base = the previous conflict's server snapshot; the `@longform` resolver preserves both edits across the chain.
- [ ] **`maxRetries` precedence:** per-type registered with `maxRetries: 10`, per-call `maxRetries: 3`, always-conflicting `use-this` → `'retries-exhausted'` with `attempts: 3` (per-call wins over per-type wins over default 5).
- [ ] **Per-resource rollback:** multi-resource batch where one resource rolls back (validation-failed) and a sibling commits → only the failed one restores from its snapshot.
- [ ] **Async resolver:** queue timeout suspended while the handler awaits; fresh timeout on re-submit.
- [ ] **Hold-pending-fanouts:** type into a `@longform` field; deliver a fanout mid-edit; assert pre-fanout keystrokes survive the merge.
- [ ] **Always-resolve:** simulate a *while-connected* network/mesh throw → resolves `infrastructure-error`, optimistic rolled back, Promise does NOT reject.
- [ ] **ontology-stale:** resolves `ontology-stale`, fires `onShouldRefreshUI`, does NOT roll back.
- [ ] **Connection-gated rollback (invariant 10):** with the mock in a non-`'connected'` state, a transaction that would otherwise time out does NOT roll back and arms no rollback; on reconnect it re-submits with the same `newETag` and resolves `committed` (server replay). Capable-of-failing: an impl that fires `timeout` while disconnected rolls back the optimistic state the banner promised was queued.
- [ ] **Reconnect snapshot held (invariant 8 extension):** a resource has pending optimistic state held through a disconnect; on reconnect the auto-resubscribe initial snapshot arrives → it is held, the flushed write merges, the pre-disconnect keystrokes survive (a non-holding impl clobbers them with the server snapshot).
