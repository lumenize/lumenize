Phase 5 (nebula-auth fire-and-forget cleanup) remains. **NOTE:** repo-wide `type-check` + `check-examples` are red ONLY from **pre-existing non-email** issues (`ts-runtime-validator` stale doc path; `fetch`/`nebula-auth-registry`-AccessEntry/`nebula`-teardown type errors ‚Äî the AccessEntry one is the parent-child session's), **not** this work. Split from the former `cloud-tests-email-and-nebula.md`; the `apps/nebula`-in-CI half is [`tasks/nebula-in-ci.md`](nebula-in-ci.md) (deferred ‚Äî consumes this).

## Goal
Two outcomes, sequenced so the milestone lands first and the breaking change is gated:
1. **Unblock the secret-less Claude-hosted CI lane (the token-drop)** ‚Äî omit the Cloudflare-remote-binding test projects when no CF creds are present, via an **opt-out** flag. **Independent of the package work; lands first.**
2. **Extract `@lumenize/email`** (MIT, near-dependency-free transport **library** ‚Äî its only dependency is the zero-dep sibling `@lumenize/debug`) and refactor `@lumenize/auth` + `@lumenize/nebula-auth` onto it, **collapsing the provider-by-subclass API into one provider-agnostic `AuthEmailSenderBase`** (provider chosen by env) ‚Äî a **breaking but cleaner** change, sequenced behind a **coordination gate** so the rip-out doesn't collide with the concurrent mainstream session.

The package is *designed* to also back a mesh `this.svc.email` capability, **deferred** to its first consumer ‚Äî see [Deferred](#deferred--later-pointers-not-built-here).

## Why this exists (context + the security story)
CI is green (memory `project_ci_cloud_tests`). The blocker for a secret-less hosted lane (Claude Code web / Dispatch) is plaintext + loggable env vars; the broad `CLOUDFLARE_API_TOKEN` is the scary one. Its **only** reason to exist in our secrets is the e2e remote-proxy ‚Äî prod sends via the `EMAIL` binding directly (no token). `RESEND_API_KEY` + `TEST_TOKEN` are low-risk + scopeable ‚Üí acceptable in plaintext; the CF token is not.

## Architecture ‚Äî the breaking extraction
**Today**: `CloudflareEmailSender`/`ResendEmailSender` `extends AuthEmailSenderBase extends WorkerEntrypoint`; provider hardcoded per deployment by *which subclass* the `AuthEmailSender` entrypoint extends. `ResolvedEmail` **and** the 5-variant `EmailMessage` union both live in `packages/auth/src/types.ts`.

**Target**:
- **`@lumenize/email` (library) owns**: `ResolvedEmail` (moved from auth ‚Äî a plain TS type, ADR-001), `EmailTransport` interface (`sendEmail(email: ResolvedEmail): Promise<void>`), `CloudflareEmailTransport` (reads `env.EMAIL`), `ResendEmailTransport` (reads `env.RESEND_API_KEY` + `fetch`), `createEmailTransport(env: object, { provider? })`. Plain classes (NOT `WorkerEntrypoint`); `env` typed `object` with narrowed optional access (`(env as { EMAIL?; RESEND_API_KEY?: string; EMAIL_PROVIDER?: string })`), not `as any`. No `cloudflare:workers` import; depends only on `@lumenize/debug` (which hides its own `cloudflare:workers` behind its `workerd` export condition).
- **`EmailMessage` (the 5-variant template union) STAYS in `@lumenize/auth`** ‚Äî it's template/auth-domain, not transport. **Only `ResolvedEmail` moves.**
- **`@lumenize/auth` (breaking)**: ONE concrete **`AuthEmailSenderBase`** (`WorkerEntrypoint`) ‚Äî keeps `EmailMessage` + the 5 templates + the `send(EmailMessage)` dispatcher; its `sendEmail()` **composes `createEmailTransport(this.env)`**. User-developers `class X extends AuthEmailSenderBase { from = ‚Ä¶ }`; **provider chosen by env.** `CloudflareEmailSender` / `ResendEmailSender` are **removed**.
- **`@lumenize/nebula-auth`**: `NebulaEmailSender extends AuthEmailSenderBase` (was `CloudflareEmailSender`); keeps `from`/`appName` + the `X-Lumenize-Auth-Instance` header override.
